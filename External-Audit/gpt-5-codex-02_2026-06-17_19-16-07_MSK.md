## Слабые стороны и findings

### REAUDIT-01 - High - Monetary closed-form не соответствует Protocol v35.26.2

Спецификация v35.26.2 теперь задает:

- `supply_moneta(W) = EMISSION_moneta * W`;
- `supply_moneta(0) = 0`;
- первая выплата происходит на `settle(1)` победителю окна 0.

Код все еще реализует старую формулу:

- `crates/mt-account/src/lib.rs:523-527`: `supply_moneta(W) = EMISSION_moneta * (W + 1)`.
- `crates/montana-node/src/commands/status.rs:131-135`: CLI печатает `emission × (W+1)`.
- `crates/mt-account/src/lib.rs:1917-1939` и `crates/mt-account/tests/determinism_invariants.rs:319-333`: тесты закрепляют `supply_moneta(0) == EMISSION`.
- `crates/mt-examples/examples/m3_account.rs:490` также документирует старую формулу.

При этом `apply_emission()` уже делает early return на `window_w == 0`, то есть state transition ближе к новой спецификации, чем helper/status/tests. Получается опасная ситуация: состояние и публичная closed-form функция расходятся, а green tests закрепляют неверный invariant.

Impact:

- Неверная observability и accounting UX в `montana-node status`.
- Conformance illusion: test suite green, но денежная формула не соответствует текущей спеки.
- Если `supply_moneta()` будет использован в sanity-check `sum(balance) == supply_moneta(W)`, узел будет ложноположительно fail-stop или, наоборот, разработчики будут вынуждены подгонять state под старое правило.

Recommendation:

- Изменить `supply_moneta(window, params)` на `params.emission_moneta * u128::from(window)`.
- Обновить все тест-векторы: `W=0 -> 0`, `W=1 -> EMISSION`, `W=100 -> 100 * EMISSION`.
- Обновить `status.rs`, examples, `AUDIT.md`, `docs/CONFORMANCE.md`, `docs/SPEC_DEVIATIONS.md`.
- Добавить conformance row/test именно для monetary closed-form, а не только для `ProtocolParams.emission_moneta`.

### REAUDIT-03 - Medium/High - FastSync anchor fix is partial, not exact anchor-binding

Previous self-block was partially fixed: when a lagging node observes a cemented proposal and sends `FastSyncRequest { anchor_window: w }`, it now inserts `ctx.recent_roots.insert(w, header.state_root)` before returning (`crates/montana-node/src/commands/start.rs:1208-1237`).

Residual gaps:

- Server decodes `req`, but serves `Snapshot::from_tables(*ctx.current, ...)` and sets response `anchor_window: *ctx.current`, not `req.anchor_window` (`start.rs:1443-1483`).
- Client checks only `chunk_anchor > current`, then calls `wire_chunk_to_sync()`.
- `wire_chunk_to_sync()` drops `anchor_window` entirely (`crates/montana-node/src/commands/fastsync.rs:16-47`).
- `FastSyncClient::finalize()` matches reconstructed root against any root in `recent_roots`, not against the exact requested anchor (`crates/mt-sync/src/client.rs:106-127`).
- `mt_net::FastSyncResponseChunk` comment says finalize should use `recent_roots[anchor_window]` instead of scanning all known roots (`crates/mt-net/src/payloads.rs:68-72`), but implementation does the opposite.

Impact:

- The original self-block for the exact observed `w` is improved, but the protocol still accepts "any recently observed root" semantics.
- A peer that has advanced beyond requested `w` may send a snapshot at another head; this can be liveness-friendly but is not the exact anchor-binding claimed by code comments and previous recommendation.
- Mixed chunks from different anchors are not rejected at the `mt_sync` layer because anchor is not retained in `FastSyncChunk`.

Recommendation:

- Store requested anchor in `FastSyncClient`.
- Require all chunks to carry the same `anchor_window`.
- Preserve `anchor_window` in `FastSyncChunk` or wrap it at node layer.
- Verify root equality against exactly `recent_roots[anchor_window]`.
- Decide explicitly in spec whether FastSync is exact-anchor or "any observed recent root"; code comments, tests and protocol text must say the same thing.

### REAUDIT-05 - Medium - Legacy non-XX Noise_PQ handshake remains public

Production path uses `NoisePqXxConfig`, but older XK-like API remains exported and tested:

- `crates/mt-noise-pq/src/lib.rs` exposes old `initiator_send_msg1`, `initiator_receive_msg2`, `responder_send_msg2`, etc.
- `crates/mt-noise-pq/src/libp2p_upgrade.rs` wraps that old path.
- `crates/mt-net-transport/src/noise_pq_upgrade.rs` exposes old libp2p upgrade wrappers.
- `crates/mt-net-transport/tests/e2e_noise_pq_with_libp2p_upgrade.rs` and `crates/mt-noise-pq/tests/libp2p_upgrade_e2e.rs` still exercise it.

The old path signs transcript hashes without the new XX post-decap shared-secret binding that Network v1.3.1 now specifies for production XX.

Impact:

- Current production builder does not use it, so this is not a direct production break.
- But the stale public API is an attractive footgun for future code, examples, or external integrators.
- It weakens the claim "production transport is only Noise_PQ XX" because the crate still ships another authenticated PQ transport path.

Recommendation:

- Remove the old upgrade modules from public exports, or gate them behind `#[cfg(test)]` / a clearly named non-production feature.
- Mark old APIs `#[deprecated]` with a message pointing to `xx_handshake`.
- If they must stay, update their transcript signatures to the same post-decap binding standard and add explicit "not production" docs.
