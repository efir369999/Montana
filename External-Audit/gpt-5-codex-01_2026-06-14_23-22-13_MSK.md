## Детальные замечания

### EXT-GEN-01: ProtocolParams не соответствует Genesis Decree v35.26.0

Спека v35.26.0 требует включить в `protocol_params`:

- `bootstrap_pow_difficulty` (`Montana Protocol v35.26.0.md:2455`)
- `max_protocol_payload_bytes` (`:2456`)
- `max_sf_ciphertext_bytes` (`:2457`)
- `n_seed` (`:2460`)
- `genesis_active_operators` (`:2461`)

Инварианты спеки говорят, что `n_seed` и `genesis_active_operators` фиксированы в коде, runtime override запрещен, pre-seed входит в genesis Merkle roots (`:2477-2479`), а сетевые параметры consensus-binding и любое отклонение reject (`:2480-2484`).

Реализация в `crates/mt-genesis/src/lib.rs` осталась на старом layout:

- `PARAMS_ENCODED_SIZE = 4094` и комментарий layout без новых полей (`:12-18`)
- `ProtocolParams` содержит только старые поля и две bootstrap pubkey (`:49-78`)
- canonical encode не пишет новые поля (`:80-103`)
- `genesis_params()` не задает `bootstrap_pow_difficulty`, payload/SF лимиты, `n_seed`, active operators (`:112-137`)

Это не косметика: текущий код не может вычислить Genesis State Hash, соответствующий v35.26.0. Любая реализация, следующая спекам v35.26.0 буквально, получит другой Genesis State Hash и другое genesis state.

**Рекомендация:** расширить `ProtocolParams` до полного v35.26.0 layout, обновить `PARAMS_ENCODED_SIZE`, byte-exact vectors, Genesis State Hash KAT и reject-тесты на mutation каждого нового поля.

### EXT-GEN-02: force_active cohort меняет genesis state вне Genesis Hash

`crates/mt-genesis/src/manifest.rs` прямо говорит, что manifest не входит в Genesis Decree и не является `genesis_state_hash binding` (`:7-14`). При этом `GenesisPeer.force_active` описан как test-cohort pre-seed в genesis NodeTable/AccountTable (`:38-54`).

Фактический runtime делает именно это:

- `LocalState::bootstrap(..., extra_actives)` принимает runtime peers (`crates/montana-node/src/state.rs:29-33`)
- каждый `force_active` peer вставляется в AccountTable и NodeTable с `chain_length=1`, `last_confirmation_window=0` (`:93-135`)
- если `accounts.bin` отсутствует, bootstrap state строится из этих runtime extras (`:145-157`)

А текущий `scripts/genesis-manifest.json` содержит active cohort через `force_active: true`: Frankfurt (`:18`), Armenia (`:28`), Vilnius (`:38`), Nicosia (`:48`). Это production-like файл, не только unit fixture.

Риск: два узла с одинаковыми embedded `ProtocolParams` и одинаковым Genesis State Hash, но разными manifest-файлами, стартуют с разными `node_root/account_root`. Это прямой split-brain vector на genesis/runtime boundary.

Дополнительно: manifest validation проверяет длину hex, но не hex alphabet; затем `hex_to_pubkey(...).expect("hex")` может panic на invalid chars. Это DoS/операционный fail-fast вместо clean reject.

**Рекомендация:** либо перенести active cohort в `ProtocolParams.genesis_active_operators`, либо жестко запретить `force_active` для production manifest. Runtime manifest должен быть только discovery metadata. Добавить KAT: одинаковый Genesis Hash должен однозначно определять начальный `state_root`.

### EXT-QRM-01: active predicate в node runtime не соответствует спекам

Спека: active node predicate считается за последние `2τ₂` (`Montana Protocol v35.26.0.md:1256-1262`). При `τ₂=20160` это 40320 окон.

В `mt-state` primitive это есть (`crates/mt-state/src/lib.rs:458-460`). Но `montana-node` использует локальную функцию `active_chain_length_at()` с:

- `const QUORUM_ACTIVE_HORIZON: u64 = 4` (`crates/montana-node/src/commands/start.rs:838-843`)
- filter `w.saturating_sub(last_seen) <= horizon` (`:872-883`)

Комментарий над функцией говорит про `2τ₂`, но код считает 4 окна. Это меняет quorum, leader acceptance, finalization pressure и BFT/liveness assumptions. Узел, который по спекам активен еще 40320 окон, в runtime выпадает из quorum после 5 окон.

**Рекомендация:** использовать `mt_state::is_active(node, w, params.tau2_windows)` для consensus quorum или явно изменить Protocol spec и Genesis Decree с protocol_version bump. Локальный live quorum можно использовать только как networking heuristic, не как consensus predicate.

### EXT-QRM-02: live-set/degraded quorum не описан в Protocol spec

Runtime quorum зависит от локального времени:

- `peer_seen` хранит `Instant` и observed interval (`crates/montana-node/src/commands/start.rs:902-904`)
- `peer_alive()` считает timeout через локальный wall-clock и genesis grace (`:953-970`)
- `live_quorum_need()` считает quorum только по self + locally alive peers (`:973-985`)
- degraded branch использует `quorum(present_cl)` вместо `quorum(active_cl)` (`:1471-1479`)

Это может быть полезным failsafe, но оно не является deterministic state rule. Спека v35.26.0 требует active set из Node Table и `last_confirmation_window`, вычисляемый независимо всеми узлами. В текущем виде разные узлы могут иметь разные `peer_seen` и разные acceptance thresholds для одного и того же proposal evidence set.

**Рекомендация:** формализовать live-set/degraded quorum в новой spec версии и зафиксировать его в Genesis Decree/protocol version, либо убрать из consensus acceptance path.

### EXT-SYNC-01: FastSync anchor integrity gate может сам себя заблокировать

`mt-sync` ожидает, что client verify snapshot root against independently observed proposal roots (`crates/mt-sync/src/lib.rs:9-14`). `FastSyncClient::finalize()` реально ищет root в `recent_roots` (`crates/mt-sync/src/client.rs:106-127`).

Но в `montana-node` lagging path делает:

- если `w - current > fast_sync_lag_threshold`, отправляет `FastSyncRequest { anchor_window: w }` (`crates/montana-node/src/commands/start.rs:1323-1333`)
- ставит `fast_sync = Some(...)` (`:1341-1343`)
- сразу `return Ok(())` (`:1347`)

Observed `header.state_root` для этого `w` не добавляется в `recent_roots`. Добавление root происходит только после локального apply sequential proposal (`:1531-1539`). В результате узел, который действительно отстал, запрашивает fast-sync ровно тогда, когда не имеет anchor root для `finalize()`.

Вдобавок `FastSyncResponseChunk.anchor_window` теряется при конвертации `wire_chunk_to_sync()` (`crates/montana-node/src/commands/fastsync.rs:16-47`), а `FastSyncClient` матчится с любым root из recent set, не с exact requested anchor. Это лучше, чем слепо принять snapshot, но хуже, чем точная anchor-binding проверка.

**Рекомендация:** перед fast-sync return сохранять observed `(w, header.state_root)` как pending anchor root без apply, передавать requested anchor в `FastSyncClient`, проверять exact `anchor_window` во всех chunks и root equality именно для requested anchor.

### EXT-MON-01: emission/supply invariant расходится на window 0

Спека: `supply_moneta(W) = EMISSION_moneta × (W + 1)` (`Montana Protocol v35.26.0.md:223`, `:1096-1101`) и каждый финализированный proposal должен иметь `delta_supply == EMISSION_moneta` (`:1082-1092`).

Код в `mt-account`:

- `apply_emission(... window_w ...)` сразу `return`, если `window_w == 0` (`crates/mt-account/src/lib.rs:561-570`)
- тесты одновременно закрепляют `supply_moneta_window_zero_is_emission` и `apply_proposal_emission_no_op_at_window_zero`

Итог: pure helper и фактический state transition расходятся для первого окна. Это monetary invariant drift: supply function говорит одно, AccountTable balances после apply W=0 дают другое.

**Рекомендация:** выбрать один canonical вариант. Если W=0 должен иметь emission, убрать early return и обновить tests. Если genesis window не имеет emission, изменить Protocol spec на `EMISSION × W` и зафиксировать это как deliberate deviation.

### EXT-VPN-01: revoke authorization broken

`handler_revoke()` строит сообщение `"montana-revoke-v1:" || address || nonce`, проверяет Ed25519 signature с `body.pubkey`, а затем удаляет `state.accounts.remove(&addr)` (`crates/mt-vpn-balance/src/main.rs:540-553`).

Нет проверки, что `body.pubkey` совпадает с `rec.pubkey_hex`, закрепленным при signed heartbeat (`:279-289`). Значит атакующий может:

1. выбрать чужой `address`;
2. создать свой Ed25519 ключ;
3. подписать revoke-message своим ключом;
4. удалить чужую запись.

Это DoS/accounting deletion для VPN ledger.

**Рекомендация:** перед remove читать account record, требовать `rec.pubkey_hex == Some(body.pubkey)`, запрещать revoke для legacy unsigned accounts либо вводить отдельный ownership proof.

### EXT-VPN-02: admin purge доверяет header и разрешает отсутствие header

`handler_purge()` проверяет только `x-real-ip` (`crates/mt-vpn-balance/src/main.rs:470-484`). Если header отсутствует, `client_ip == ""`, и условие запрета не срабатывает. Если сервис доступен не строго за trusted reverse proxy, любой caller может вызвать purge. Если proxy пропускает spoofed header, caller может отправить `x-real-ip: 127.0.0.1`.

Purge удаляет только inactive zero-balance accounts (`:485-504`), но это все равно unauthenticated admin action.

**Рекомендация:** закрыть route auth token/mTLS/Unix socket/admin bind, reject missing proxy headers, принимать proxy headers только от trusted proxy IP.

### EXT-VPN-03: heartbeat trust boundary и public-by-code subscription

Heartbeat trust:

- client IP берется из `x-real-ip` или `x-forwarded-for` (`crates/mt-vpn-balance/src/main.rs:195-205`)
- новые accounts могут использовать legacy unsigned mode (`:243-268`)
- downgrade запрещен только после TOFU-pinning (`:279-305`)
- bind default `127.0.0.1:5009` (`:600-604`), что снижает риск при правильном reverse proxy, но не защищает при прямой публикации.

Subscription:

- `/vpn/sub` возвращает hardcoded VLESS UUIDs, Reality `pbk`, `sid` и entry domain (`:410-430`)
- route публично подключен (`:590-596`)

Если это намеренно публичная subscription endpoint, это ожидаемо. Если доступ должен быть ограничен владельцами accounts, текущий код выдает usable credentials всем.

**Рекомендация:** явно определить trust boundary. Для non-public режима добавить signed account auth на `/vpn/sub`, per-account issued credentials, rotation/revocation, и не принимать proxy headers от внешнего клиента.

### EXT-FFI-01: bindings копируют секреты без zeroize

Core `mt-crypto` аккуратно защищает `SecretKey`, но bindings повторно создают обычные stack/Vec copies:

- C FFI: `master_arr`, seed arrays, `master`, `acc_seed`, secret output path без zeroize (`crates/mt-bindings/src/ffi_c.rs:65-69`, `:84-91`, `:126-145`)
- JNI: `master`, `acc_seed`, `sk`, `buf` с `pk||sk||account_id`, `sk_bytes` из Java (`crates/mt-bindings/src/ffi_jni.rs:90-108`, `:123-139`)

Это не remote exploit, но для mobile wallet/recovery forensic model это important memory hygiene issue.

**Рекомендация:** использовать `zeroize::Zeroizing` / explicit `zeroize()` для all seed/sk buffers, избегать long-lived `Vec` с secret key, документировать ownership и wipe contract на C/JNI/WASM boundary.

### EXT-NOISE-RESIDUAL: ранее известный Noise_PQ transcript-binding gap

Текущий `xx_handshake.rs` по-прежнему подписывает transcript, включающий ephemeral KEM pubkeys/ciphertexts/identity pubkeys, но не post-decap `ss_i/ss_r` в signature input. Master key включает `ss_i || ss_r || transcript` (`crates/mt-noise-pq/src/xx_handshake.rs:26`, `:383-386`), но signatures строятся до добавления shared secrets в подписываемую область (`:189-200`, `:255-270`, `:297-305`, `:353-366`).

Это было найдено в предыдущем внешнем отчете как residual. Не считаю это новой primary finding текущей delta-фазы, но оно остается важным для формального AKE proof.

**Рекомендация:** либо обновить spec/proof с явным обоснованием текущего порядка, либо подписывать transcript hash после decap/encap binding, например через confirmation MAC/signature over derived handshake hash.

## Рекомендации аудитора

1. Немедленно заморозить audit artifact: clean git tree, exact tag, exact spec versions. Не продолжать mainnet-readiness claim на dirty tree.
2. Сделать Genesis Decree единственным источником consensus genesis state: включить сетевые лимиты и весь active cohort в `ProtocolParams`; удалить production `force_active` из runtime manifest.
3. Добавить KAT: `canonical_encode(protocol_params)` byte-exact, `genesis_state_root` byte-exact для 1-node и 5-node cohort, reject на перестановку/дубликаты N_SEED.
4. Убрать `QUORUM_ACTIVE_HORIZON=4` из consensus acceptance или оформить как новую protocol version. Consensus active predicate должен быть deterministic и spec-bound.
5. Переписать FastSync client/server anchor handling: exact requested anchor, persisted observed root, chunk anchor consistency, root equality for that anchor.
6. Исправить monetary invariant: либо начислять emission на W=0, либо изменить spec и tests на `supply(W)=EMISSION×W`.

## Итоговая оценка безопасности

**Оценка: 5/10 для текущего состояния code-on-disk.**

Низкоуровневые криптографические и state primitives выглядят существенно сильнее этой оценки: там хорошее покрытие тестами, KAT, deterministic checks и аккуратные доменные разделители. Но текущий runtime/protocol boundary не готов к безопасному внешнему запуску: Genesis Decree не соответствует v35.26.0, active cohort может задаваться вне Genesis Hash, quorum/active rules не совпадают со спецификацией, FastSync integrity gate может не работать для реально отставшего узла, а VPN accounting API содержит authorization bugs.

Если исправить EXT-GEN-01/02, EXT-QRM-01/02 и EXT-SYNC-01, базовая оценка может подняться примерно до 7/10 до следующего аудита. До этого момента статус должен быть: **не mainnet-grade, release blocked по spec-code compliance и genesis consensus binding**.
