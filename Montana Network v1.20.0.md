# Montana — Network Layer Specification

**Version:** 1.20.0 (2026-07-06)

**Layer:** Network — sits between Protocol (low) and App (high).


---

## Introduction

Montana's network layer covers transport and discovery between consensus nodes and clients. This specification historically lived as inline sections inside Montana Protocol; it has been split into its own file to separate the layers per the [I-7] minimal-cryptographic-surface principle and to make independent audit easier.

**What this spec covers:**

- Transport layer via libp2p (TCP + Noise_PQ XX → Yamux), where Noise_PQ XX is the post-quantum security upgrade replacing the classical TLS 1.3 + Noise XK chain
- Traffic obfuscation (TLS-mimicry, ECH, padding, timing)
- Identity-Bound Tunnel (IBT) — proof that a peer holds the privkey corresponding to its node_id
- Transport Randomness — unpredictable network-bound seeds
- PeerRecord and discovery
- Mesh Transport (Bluetooth / Wi-Fi Direct, store-and-forward)
- Sync protocols (FastSync, BatchLookup)
- Network-layer Threat Model
- Network-layer binding KAT vectors
- apply_mesh_frame and apply_store_and_forward — normative rules
- Final gate audit

**What this spec does NOT cover:**

- State machine, apply_proposal, operations (Transfer / OpenAccount / ...) — see Montana Protocol
- Crypto primitives (ML-DSA-65, ML-KEM-768, SHA-256, PBKDF2 / HKDF) — see Montana Protocol, Cryptography section
- UI / Wallet / Messenger / Channels / Contacts / Profile / Junona / Browser — see Montana App

## Network layer

All time parameters at the network layer (frame rate, padding window, feeler interval, Dandelion timers) are implementation guidance for the node's local network stack. They operate on the node's local clock and are outside the scope of consensus state.

### Transport obfuscation

Montana is a personal network. Each node is the participant's personal server. The transport layer is built from this definition: a personal server answers only to participants, a personal messenger hides message timing, personal = affordable to an ordinary person.

#### Encryption

All P2P connections are encrypted by Noise_PQ XX (ML-KEM-768 ephemeral KEM on both sides + ML-DSA-65 identity + ChaCha20-Poly1305 AEAD), the production transport handshake. Inbound listeners default to TCP port 8444. The content of the traffic is not accessible to an observer.

#### Identity-Bound Tunnel (IBT)

A personal server answers only to network participants. After the Noise_PQ XX handshake completes, the client sends an authentication proof. Nodes (registered and invited) sign with the node keypair. Accounts (clients) sign with the account keypair.

```
proof = ML-DSA-65_sign(client_privkey,
          "mt-tunnel-online" || server_node_id || floor(current_window_index / 2)
          || online_session_nonce)

where:
  online_session_nonce  32B    — generated by the client from CSPRNG for each
                                 handshake, transmitted in the plain part of the IBT
                                 advertisement alongside the proof
```

The server checks:

1. Signature is valid for the claimed client_pubkey
2. Window slot = current OR previous (window = 2 window_index)
3. Access level — the server checks client_pubkey against three tables in order, the first match determines the level:
   - `node_id = SHA-256("mt-node" || client_pubkey)` in Node Table → **full gossip** (client connected with the node keypair)
   - `node_id` with `node_pubkey = client_pubkey` in Candidate Pool → **read-only gossip**: receives proposals (candidate connected with the node keypair)
   - `account_id = SHA-256("mt-account" || suite_id || client_pubkey)` in Account Table → **connection to a trusted node** (client connected with the account keypair)
   - None matched → reject

Conditions 1-2 met + access level from step 3 determined → Noise handshake → Montana's P2P network at the corresponding access level.
Any condition failed → the node aborts the Noise_PQ XX session and closes the connection. This is standard server behaviour for mandatory client authentication — there are millions of such servers on the internet (corporate portals, APIs, banking systems).

Replay protection: three-layer defence simultaneously. (a) `server_node_id` binds the proof to a specific recipient — replay against a different server is impossible. (b) The window slot caps the replay window at 2 windows (≤120 seconds at genesis calibration) — the proof becomes invalid after one window. (c) **Session nonce tracking.** The server keeps `used_online_nonces[client_pubkey]` — a set of `online_session_nonce` values used by this client within the current / previous window. On receiving a proof with `online_session_nonce ∈ used_online_nonces[client_pubkey]` → reject (replay within the window slot). Set pruning: entries older than 2 windows are removed (nonce reuse is acceptable after expiry — the window slot is no longer valid). This is a defence-in-depth closure of the MITM-replay class inside the window slot — even if a proof is intercepted, the attacker cannot reuse it against the same server within those 2 windows.

**Verification procedure for an online peer.** On receiving an online IBT advertisement, the server node performs the checks in the following order (mirror of the mesh IBT procedure):

1. Parse the advertisement; extract `client_pubkey`, `online_session_nonce` (32 B), `proof` (3309 B ML-DSA-65 signature).
2. Check that the ML-DSA-65 signature is valid for `client_pubkey` over the message reconstruction `"mt-tunnel-online" || server_node_id || u64_LE(floor(current_window_index / 2)) || online_session_nonce` (current window slot). If it does not match — try with `floor(current_window_index / 2) - 1` (previous slot).
3. Check `server_node_id == local_node_id` (proof bound to this server).
4. Check `online_session_nonce ∉ used_online_nonces[client_pubkey]` (replay block).
5. Look up `client_pubkey` against three tables in order: Node Table (via `SHA-256("mt-node" || client_pubkey)` lookup) → Candidate Pool → Account Table (via `SHA-256("mt-account" || suite_id || client_pubkey)`). The first match determines the access level. None matched — reject.
6. All checks passed → accept; add `online_session_nonce` to `used_online_nonces[client_pubkey]`; start the Noise handshake; start the P2P session at the access level from step 5.
7. Any check failed → the node aborts the Noise_PQ XX session and closes the connection. Silent reject is optional (do not give an attacker feedback about which step failed).

**`used_online_nonces` memory bound.** Set per `client_pubkey`, ephemeral, transport-layer (not consensus state — [I-14] formally does not apply). DoS bound on two levels: (i) per-pubkey set bounded by `MAX_ONLINE_NONCES_PER_PUBKEY = 256` (an attacker with one keypair cannot flood more than 256 nonces per window slot — the handshake rate-limit already caps the real stream below < 256 / 2τ₁); (ii) global bound `MAX_ONLINE_NONCES_TOTAL = 65 536` (32 B × 65536 ≈ 2 MB per server — acceptable memory on commodity hardware [I-5]). Pruning at the window-slot boundary: entries older than 2 windows are removed automatically. Defence against many distinct client_pubkeys: server-side handshake rate-limit (see backpressure rule [B5] in the spec — `max_pending_requests_per_peer`) caps the new-handshake stream per source.

Discovery-seed exception: a small set of discovery-seed peers is hard-coded as `(IP, node_id, pubkey)` so that a newcomer with no peers can find the mesh. A discovery seed accepts a proof from any valid ML-DSA-65 key (the Account Table is not consulted), since it serves nodes that have not yet synchronised any state. Connection flooding against a discovery seed is bounded by the same transport-layer handshake rate-limit that protects every peer (backpressure rule [B5], `max_pending_requests_per_peer`): a discovery seed caps new handshakes per source and silently drops excess.

**Mesh transport IBT extension.**

Mesh transport (see the «Mesh Transport» subsection below) operates without a fresh `window_index` — the device may be offline for hours or days until the next sync with the internet-side network. The IBT proof in the mesh context uses a **cached** `window_index` — the last known window value from any previous online connection.

Formula for mesh transport:

```
mesh_proof = ML-DSA-65_sign(
    client_privkey,
    "mt-tunnel-mesh"
    || peer_node_id
    || floor(cached_window_index / 2)
    || mesh_session_nonce)

where:
  cached_window_index    u32    — the last known window value from any
                                  previous online handshake or
                                  gossiped proposal
  mesh_session_nonce     32B    — generated by the handshake initiator
                                  from CSPRNG, transmitted in the plain
                                  part of the mesh advertisement
```

**Acceptable staleness bound.** The peer accepts `cached_window_index` in the range `[peer.known_window_index - 7 × τ₁, peer.known_window_index]`. Above `7 × τ₁` the cached value is treated as too stale — the peer rejects the mesh IBT handshake and requires a fresh value through any available channel before continuing.

**Session nonce tracking.** The peer keeps `used_nonces[sender_pubkey]` — a set of `mesh_session_nonce` values used by this sender within the acceptable staleness window. On receiving a proof with `mesh_session_nonce ∈ used_nonces[sender_pubkey]` → reject (replay). Set pruning: entries older than `7 × τ₁` are removed (nonce reuse after expiry is acceptable — the cached_window_index is no longer valid).

**The domain separator MUST be `mt-tunnel-mesh`, not `mt-tunnel-online`.** A separate separator is critical — otherwise an attacker who intercepted an online IBT proof (window slot = `2 × τ₁` replay) could reuse it in the mesh context, where the staleness window is widened to `7 × τ₁`. Cross-context replay is blocked at the domain-separation level.

**Replay surface analysis.**
- Online IBT (separator `mt-tunnel-online`): replay window `2 × τ₁` — narrow, plus per-nonce tracking blocks replay inside the window slot (see Replay protection above).
- Mesh IBT (separator `mt-tunnel-mesh`): replay window widened to `7 × τ₁`, but replay is blocked by per-nonce tracking.
- Cross-context: domain separation makes a proof for one context invalid in the other.

**Verification procedure for a mesh peer.**
1. Parse the advertisement; extract `sender_pubkey`, `mesh_session_nonce`, `proof`.
2. Verify the signature against `sender_pubkey`.
3. Recover `cached_window_index` from the proof message reconstruction (the peer knows sender_pubkey, peer_node_id is known locally; the peer tries the range `[known_window_index - 7 × τ₁, known_window_index]` until a matching value is found; if no match — reject).
4. Check `mesh_session_nonce ∉ used_nonces[sender_pubkey]`.
5. All ok → accept; add `mesh_session_nonce` to used_nonces; start the mesh session.
6. Any check failed → silent reject (no error message, to avoid giving an attacker feedback).

#### Uniform Framing

All Montana messages inside an IBT connection are fragmented into fixed-size frames:

```
frame_size = 1024 bytes

frame format:
  flags     1B    (0x01 = data, 0x02 = padding, 0x04 = continuation)
  length    2B    (payload length, ≤1021B)
  payload   1021B (data or random padding up to frame_size)
```

A personal messenger hides timing: a continuous stream of frames flows between nodes. Real Montana messages replace padding frames rather than adding to them. An on-network observer cannot distinguish a transfer from a proof of time from silence — everything is the same encrypted frames.

Parameters:

- Baseline frame rate: 1 frame/s on outgoing connections. Inbound — frames when data is available
- Maximum burst: ≤ 8 frames in a row without a ≥ 10 ms pause
- Minimum padding ratio: ≥ 20% of frames within a sliding τ₁ window on outgoing
- Frame schedule: the inter-frame interval is drawn from a bounded distribution around the baseline rather than a fixed metronome, so the outgoing stream presents no constant-rate signature to a flow classifier. The jitter is bounded — a real data frame is never held longer than one baseline interval — so the Tier 1 latency target is preserved.

Personal = affordable: 24 outgoing × 1 frame/s × 1024 bytes = 24 KB/s ≈ 60 GB/month. Acceptable for a home server.

#### Transport Randomness

All randomized decisions at the transport layer (stem routing, frame scheduling, nonce generation) use a CSPRNG seeded from the OS entropy pool. A deterministic PRNG derived from node state is forbidden for transport-layer randomness. Frame scheduling draws its inter-frame intervals from this CSPRNG so the outgoing stream carries no fixed-rate signature; the distribution is bounded so real data is never delayed beyond one baseline interval.

Transport obfuscation is orthogonal to consensus. The TimeChain and state machine work over any transport without changes.

#### Post-quantum transport

The production transport is a single post-quantum handshake: Noise_PQ XX with ML-KEM-768 ephemeral keypairs on both sides of the handshake, ML-DSA-65 identity signatures over the transcript, and ChaCha20-Poly1305 AEAD framing on the established session. Transport confidentiality is post-quantum end-to-end, and no classical Diffie-Hellman is present in the protocol layer; the DPI-obfuscation role is carried by uniform framing on top of Noise_PQ XX. PeerId is derived from each peer's ML-DSA-65 identity public key as the SHA-256 multihash (libp2p / IPFS sha2-256 multihash code 0x12), so cryptographic and routing identities are bound to the same key material.

**Multi-confirmer cementing protocol.** A network with a single Active operator runs in a singleton-cementing regime: the sole Active operator proposes and confirms alone, since a quorum over a single Active node is that node itself (quorum = 1). This is the bootstrap period that follows the empty genesis window — the first self-admitting operator cements its own chain through the ordinary rules, with no special baked identity. The multi-confirmer cementing protocol formalised below is the normative path for operational regimes in which additional operators accumulate non-negligible `chain_length` over many τ₂ epochs.

Protocol steps per window `W`:

1. **Candidate Proposal.** The canonical proposer (per the lottery selection of `W − 2`) constructs `ProposalHeader` with `included_bundles_root` set to `H(my_own_BC)` and broadcasts the header (`MsgType::Proposal`, 0x22).
2. **Follower confirmation.** Each Active operator on receipt of the candidate Proposal (a) verifies the proposer is canonical for `W`, (b) signs its own `BundledConfirmation(W, my_endpoint, my_op_hashes, my_reveal_hashes)` where `my_endpoint = SHA-256("mt-lottery" || T_r(W) || cemented_bundle_aggregate(W-2) || my_node_id || W)`, (c) broadcasts the BC envelope (`MsgType::BundledConfirmation`, 0x20) to the proposer and to all connected peers.
3. **Proposer accumulator.** The proposer maintains a per-window BC accumulator. On each incoming BC, the proposer validates per `validate_bundle` and, if valid, adds the BC to the accumulator. The accumulator window closes when `cemented_sum = Σ node.chain_length` over all collected confirmers reaches `quorum(active_chain_length) = ⌈67 × Σ active_chain_length / 100⌉`, or when one `τ_1` window elapses since the candidate Proposal was broadcast (whichever first).
4. **Cemented Proposal broadcast.** The proposer builds the final ProposalHeader with `included_bundles_root` = sparse Merkle root over the collected BC set, and broadcasts the cemented envelope as `MsgType::Proposal` (0x22). The cemented envelope is extended from the legacy 3722-byte header-only layout to a length-prefixed schema `[header (3722 B)][u16 bundle_count][bundle_count × BC]`, where each BC is variable-length per the standard `BundledConfirmation::encode` (node_id 32 B + endpoint 32 B + window_index 8 B + u16 op_hashes count + N × 32 B + u16 reveal_hashes count + N × 32 B + signature 3309 B).
5. **Follower cementing.** Each follower on receipt of the cemented Proposal (a) parses the bundle set, (b) calls `validate_bundle` on each BC with `expected_endpoint = SHA-256("mt-lottery" || T_r(W) || cemented_bundle_aggregate(W-2) || bc.node_id || W)`, (c) builds `ProposalSettle { window_w: W, winner_id, cemented_confirmers: [all signers] }`, (d) calls `apply_proposal` with the multi-confirmer set.

The singleton-cementing path (a single confirmer, the regime of a network with one Active operator) is the special case of the protocol above with `bundle_count = 1`; the 3722-byte header-only envelope handles that special case. Operational regimes with several operators accumulating `chain_length` require the extended length-prefixed schema and validation of `bundle_count ≥ 1`.

T_r consistency requirement. The follower's `compute_endpoint` requires the canonical `T_r(W)`. Two implementation paths are viable: (i) followers tick the SSHA locally in lockstep with wall-clock and cache the per-window `T_r` history during catch-up; (ii) the proposer extends the cemented Proposal envelope with `T_r(W)` (32 B) so followers do not need to compute it locally. Path (ii) is the cleaner architectural choice; it adds 32 B per envelope and removes any drift hazard for followers whose SSHA tick is behind the cemented head.

The wire format and KAT vectors for the cemented Proposal envelope with `bundle_count ≥ 2` are specified in this spec.

**[I-1] compliance.** The entire protocol stack is post-quantum end-to-end: consensus signatures via ML-DSA-65, application-layer encryption via ML-KEM-768, and the transport handshake via Noise_PQ XX (ML-KEM-768 + ML-DSA-65). The transport stack is `TCP → Noise_PQ XX → Yamux`, with uniform framing preserved on top for DPI obfuscation.

**Wire format (normative, production XX).** The Noise_PQ XX handshake is exactly three messages. Identifiers in formulas below match the reference implementation in `crates/mt-noise-pq/src/xx_handshake.rs`. The libp2p multistream-select protocol identifier for the handshake is **`/montana/noise-pq-xx/1.0.0`** — this string is the authoritative protocol name used to negotiate the upgrade between two peers.

| Message | Size (B) | Fields |
|---------|---------:|--------|
| msg1 (initiator → responder) | 1184 | `ke_pk_i` (ML-KEM-768 pk, 1184 B) |
| msg2 (responder → initiator) | 7533 | `ke_pk_r` (1184 B) ‖ `ct_i` (ML-KEM-768 ct to `ke_pk_i`, 1088 B) ‖ `rs_id_pk` (ML-DSA-65 pk, 1952 B) ‖ `sig_r` (ML-DSA-65 sig, 3309 B) |
| msg3 (initiator → responder) | 6349 | `ct_r` (ML-KEM-768 ct to `ke_pk_r`, 1088 B) ‖ `is_id_pk` (1952 B) ‖ `sig_i` (3309 B) |

Transcript hash (input to both `sig_r` and `sig_i`) is the byte concatenation of msg1 plus the msg2-prefix-without-`sig_r` (for `sig_r`), or msg1 ‖ full-msg2 ‖ msg3-prefix-without-`sig_i` (for `sig_i`). Each signature input is domain-separated with `mt-noise-pq-xx-v1-sig-r` and `mt-noise-pq-xx-v1-sig-i` respectively. Each signature additionally binds the post-decap ML-KEM-768 shared secrets that both parties hold at signing/verification time: `sig_r` signs `transcript ‖ ss_i`, `sig_i` signs `transcript ‖ ss_i ‖ ss_r`. Identity authentication therefore covers the derived session key (AKE binding), not the transcript alone; the message field layout is unchanged.

Session keys are derived by domain-separated SHA-256 over the concatenation of both ML-KEM-768 shared secrets and the full transcript:

```
master    = SHA-256("mt-noise-pq-xx-v1-master" ‖ ss_i ‖ ss_r ‖ transcript)
sk_i_to_r = SHA-256("mt-noise-pq-xx-v1-i2r"    ‖ master)
sk_r_to_i = SHA-256("mt-noise-pq-xx-v1-r2i"    ‖ master)
```

`sk_i_to_r` and `sk_r_to_i` are 32-byte keys for ChaCha20-Poly1305 AEAD; the AEAD-wrapped byte stream is exposed as `mt_noise_pq::stream::NoisePqStream`.

**Handshake length obfuscation (normative).** Each of the three handshake messages is framed on the wire as `[total_len: u16 little-endian][core bytes][pad bytes]`, where `core bytes` is the message at its floor size from the table above (1184 / 7533 / 6349) and `pad bytes` is `pad_len` random bytes with `pad_len` drawn uniformly from `[0, HANDSHAKE_MAX_PAD]`, `HANDSHAKE_MAX_PAD = 1024`, from the operating-system CSPRNG; `total_len = core_size + pad_len`. The receiver reads the two-byte prefix, rejects any connection whose `total_len` lies outside `[core_size, core_size + HANDSHAKE_MAX_PAD]`, reads `total_len` bytes, and discards the trailing `pad_len` bytes; `core_size` is the fixed floor size of the message being received, known a priori from the table. The padding is not covered by the transcript or by either identity signature — both sign the core only, per the transcript definition above — so length obfuscation never changes the derived session keys. The two-byte prefix is a per-connection length, not a constant, so the first bytes carry no fixed magic. Within a single connection a passive observer sees no fixed length; because padding only adds, the floor of the observed length distribution equals `core_size`, so the core-size triple is recoverable only across many observations of the same profile, not from one. This removes the single-observation fixed-size fingerprint. The residual — that any post-quantum handshake is large and high-entropy, and that its floor is recoverable across many observations — is not exposed to a DPI adversary by construction: in a DPI-censoring vantage the baseline profile is T1 or higher (see «Auto-steering»), where the handshake rides inside a genuine TLS/QUIC carrier to a real cover host and its size and entropy are indistinguishable from ordinary TLS application data; the bare T0 stream is used only in vantages where no DPI-fingerprinting adversary is present.

**PeerId derivation (non-libp2p-standard).** PeerId is derived as the SHA-256 multihash (libp2p / IPFS sha2-256 multihash code 0x12) over the raw byte representation of each peer's ML-DSA-65 identity public key. This is **not** the libp2p-standard PeerId derivation, which is a multihash of the protobuf-encoded `PublicKey` message with one of the libp2p built-in key types (`RSA`, `Ed25519`, `Secp256k1`, `ECDSA`). The non-standard derivation is intentional — Montana uses ML-DSA-65 as the cross-network identity per [I-1], which is not one of libp2p's built-in key types, and the protobuf wrapper would add about fifty bytes of overhead without carrying additional security. Two implications follow: (a) a vanilla libp2p node cannot recover the public key from a Montana PeerId, and (b) Montana nodes do not accept libp2p-standard PeerIds because they have no ML-DSA-65 key to verify them with.

**Alternative XK handshake form** (`/montana/noise-pq/1.0.0`). The production transport uses the XX form above; the XK form is documented here for completeness. Its wire sizes are msg1 2272 B, msg2 6349 B, msg3 5261 B, and it requires the initiator to know the responder's static KEM public key a priori, which makes it unsuitable for the libp2p plug-in auth-upgrade slot.

```
msg1  (initiator → responder)        2272 B
  ke_pk        1184 B    initiator ephemeral ML-KEM-768 public key
  ct_rs        1088 B    KEM ciphertext encapsulated to the responder's
                         static ML-KEM-768 public key (rs_kem_pk, known a
                         priori to the initiator via the IBT directory)

msg2  (responder → initiator)        6349 B
  ct_e         1088 B    KEM ciphertext encapsulated to ke_pk
  rs_id_pk     1952 B    responder static ML-DSA-65 identity public key
  sig_r        3309 B    ML-DSA-65 signature by rs_id over the
                         signature-domain hash
                         SHA-256("mt-noise-pq-v1-sig-r" ‖ ke_pk ‖ ct_rs ‖ ct_e)

msg3  (initiator → responder)        5261 B
  is_id_pk     1952 B    initiator static ML-DSA-65 identity public key
  sig_i        3309 B    ML-DSA-65 signature by is_id over the
                         signature-domain hash
                         SHA-256("mt-noise-pq-v1-sig-i" ‖ ke_pk ‖ ct_rs ‖ ct_e
                                                      ‖ rs_id_pk ‖ is_id_pk)
```

Both sides derive identical directional session keys (each 32 B) and a 32 B transcript hash exposed to higher layers as a channel-binding token:

```
master         = SHA-256("mt-noise-pq-v1-master" ‖ ss_rs ‖ ss_e
                                                ‖ ke_pk ‖ ct_rs ‖ ct_e
                                                ‖ rs_id_pk)
sk_i_to_r       = SHA-256("mt-noise-pq-v1-i2r" ‖ master)
sk_r_to_i       = SHA-256("mt-noise-pq-v1-r2i" ‖ master)
transcript_hash = SHA-256("mt-noise-pq-v1-transcript" ‖ ke_pk ‖ ct_rs ‖ ct_e)
```

where `ss_rs = mlkem_decap(rs_kem_sk, ct_rs)` and `ss_e = mlkem_decap(ke_sk, ct_e)` on the receiving side respectively, and the corresponding `mlkem_encap(rs_kem_pk) → (ct_rs, ss_rs)` and `mlkem_encap(ke_pk) → (ct_e, ss_e)` on the sending side. Implicit-rejection semantics of FIPS 203 §6.3 are reconciled by the identity-signature check: a maliciously substituted ciphertext yields a different shared secret on the receiver, the transcript master diverges from the sender's, and the identity signature check fails (the receiver returns `BadResponderSignature` or `BadInitiatorSignature`).

**Post-handshake AEAD framing.** After the 3-message handshake completes, every application-layer byte stream between the peers is encrypted with ChaCha20-Poly1305 using the derived directional session keys. Wire format per direction:

```
direction = initiator → responder uses sk_i_to_r
direction = responder → initiator uses sk_r_to_i

per frame (each application message):
  length_be     2 B    big-endian u16, total ciphertext length including
                       16-byte Poly1305 tag (max 65 535 → max plaintext
                       65 519 = u16::MAX − tag)
  ciphertext+tag        ChaCha20-Poly1305 encryption of plaintext with
                        nonce = 0x00000000 ‖ u64_be(counter), 12 bytes;
                        counter starts at 0 and increments by 1 per
                        outgoing frame in each direction independently;
                        AAD = empty (none)
```

The 64-bit nonce counter is monotonic per direction and guaranteed not to overflow at any realistic message rate (2^64 frames at 1 Gbit / s with 64 KiB frames ≈ 2^48 years). Implementations MUST abort the connection (treat as protocol error) if the counter would overflow.

Application-layer messages larger than 65 519 bytes are fragmented by the caller (e.g. by libp2p's stream multiplexer); the AEAD layer makes no provision for in-band fragmentation.

**Capability negotiation.** The primary negotiation mechanism is libp2p multistream-select: each peer advertises the set of upgrade protocols it supports (`/montana/noise-pq/1.0.0` for the Noise_PQ handshake defined here, `/noise` or equivalent for the classical fallback) and the connection negotiates the highest mutually supported one. This is standard libp2p convention and requires no additional wire fields.

A 1-byte `pq_transport_version` field is **reserved but not currently consumed** by the IBT advertisement layout — it is held in reserve for a future explicit out-of-band signal if multistream-select proves insufficient (for example, if some operator policy needs to express «I support Noise_PQ but refuse to fall back to classical»). Values when the field becomes active:

| `pq_transport_version` | Meaning |
|---|---|
| `0x00` | Classical TLS 1.3 + Noise XK (historical, not used) |
| `0x01` | Noise_PQ v1 (this spec, ML-KEM-768 + ML-DSA-65, no hybrid X25519) |

Future versions (`0x02`, …) may add hybrid X25519+ML-KEM, larger PQ KEMs, or other refinements; the field reserves the negotiation surface.

**Cross-implementation conformance.** Reference test vectors in `crates/mt-noise-pq/tests/kat.rs` fix the responder's static ML-KEM-768 keypair seed (`byte_repeat(0x42, 64)`) and the two ML-DSA-65 identity seeds (`byte_repeat(0x77, 32)` for responder, `byte_repeat(0xAA, 32)` for initiator). The 3-message handshake on those fixed inputs must produce byte-identical `sk_i_to_r`, `sk_r_to_i`, and `transcript_hash` across implementations. Per-message wire bytes will differ run to run because the encapsulation step uses fresh OS randomness per FIPS 203 §6.2; only the derived session material is byte-exact deterministic for cross-implementation verification.

#### Transport profile ladder

The Noise_PQ XX handshake and its uniform-framing layer define the inner transport: the post-quantum session and the DPI-resistant frame stream are invariant across every deployment. The outer carrier — the byte pattern a passive observer sees before the inner handshake begins — is a negotiable profile. A node advertises the profiles it supports; a connecting peer selects the highest-preference profile its local network path permits, established empirically through the reachability-sensing mechanism below.

| Profile | Outer carrier | Observable to a DPI engine | Applicability |
|---------|---------------|----------------------------|---------------|
| T0 | none — Noise_PQ XX directly over TCP/8444 | a TCP stream of uniform 1024 B frames | default; lowest latency |
| T1 | TLS 1.3 carrier bound to a real, reachable cover host — the inner bytes travel inside a TLS record stream whose ClientHello and SNI reproduce a rotating set of live cover hosts, with the ClientHello optionally fragmented across TCP segments to defeat stateless SNI inspection; a connection that does not present the Montana inner handshake is relayed to the real cover host, so an active prober receives a genuine response and observes an ordinary reverse proxy | a TLS 1.3 session to an ordinary HTTPS host, indistinguishable from that host under active probing | a path that passes TLS but filters raw high ports, including one that actively probes suspected endpoints |
| T2 | HTTP/WebSocket over TLS through a content-delivery network | a TLS session to a large CDN endpoint | a path that filters by destination IP — the CDN address set carries unrelated tenants |
| T3 | a general-purpose pluggable-transport tunnel that carries the inner stream over an external circuit, so the node's own address is absent from the carrier the censor observes | the cover traffic of the chosen pluggable transport | a path performing both address and protocol filtering |
| T4 | mesh radio — BLE / Wi-Fi Direct, see Mesh Transport | radio advertisements at physical range | a complete internet shutdown |

The inner Noise_PQ XX session, its identity authentication through IBT, and the uniform-framing obfuscation are identical under every profile; a profile changes only the outer carrier, never the post-quantum confidentiality or the identity binding. T1, T2 and T3 introduce an external dependency — a live cover host, a CDN tenant, a pluggable-transport bridge — which an operator enables by deliberate configuration; each dependency carries its operational trust note in the Threat Model. Profile negotiation reuses the libp2p multistream-select surface and the reserved `pq_transport_version` field, and leaves the inner handshake wire format unchanged.

**Active-probe relay (T1).** A T1 responder binds to a specific cover host it can currently reach and holds a live TLS path to it. It separates a Montana peer from a probe by the inner handshake: a connection that completes the outer TLS layer and then presents the Noise_PQ XX first message is served as a Montana session; any other connection — ordinary HTTPS, a scanner, an active prober speaking TLS but no valid inner handshake — is relayed byte-for-byte to the real cover host and that host's response returned unchanged. An active prober therefore reaches a genuine site and sees the responder as an ordinary reverse proxy for that host, with no Montana-distinguishing response to confirm a suspected endpoint. The cover-host set rotates so no single host name is a durable selector, and a responder advertises only a cover host it can currently reach — a dead cover host would make the relay path fail closed and expose the profile, so reachability of the cover host is a precondition for offering T1. This external dependency places T1 in the operational-trust class of T2 and T3 (Threat Model → transport profile external dependencies).

Transport profiles are local network-stack behaviour on the node's own clock and sit outside the scope of consensus state.

### Peer selection

An open entry with a sequential-SHA-256 barrier makes Sybil nodes expensive: each Sybil = τ₂ windows of sequential SHA-256 (not parallelizable) + a selection event. Peer selection uses diversity constraints from protocol-level data (start_window) and network-level data (/16, ASN).

P2P gossip — only registered and invited nodes (IBT levels 1-2, see Transport Obfuscation → Identity-Bound Tunnel). Accounts (IBT level 3) interact through their trusted node.

#### Outgoing connections

24 outgoing, all full. Uniform framing hides message types — separate relay-only connections are not needed.

Selection: random 50/50 from the «new» and «verified» tables. Bucketing with the node's secret key. No preference by chain_length — selection is uniform.

#### Four diversity constraints

Every outgoing connection is checked against all four constraints:

```
Network:
  /16  — at most 1 outgoing per /16 subnet (IPv4) or /48 (IPv6)
  ASN  — at most 2 outgoing per autonomous system

Protocol:
  start_window — at most 2 outgoing to nodes with start_window inside one τ₂
```

Network constraints: /16 and ASN diversity. The protocol-level constraint start_window is canonically available from the Node Table.

Consequence: a Sybil cluster all registered in one τ₂ → at most 2 of 24 slots. An eclipse requires nodes in 7+ different ASes in 7+ different /16s with registrations in 7+ different τ₂.

The ASN map is loaded at startup. Without the map — fallback to /16.

#### Address manager

Two tables:

- **New** — addresses received via peer exchange and DHT. The node has not yet connected
- **Verified** — addresses to which the node has successfully connected through IBT

Bucketing: `bucket = Hash(secret_key, source_group, addr_group) % N`. Deterministic with the secret key — an attacker cannot predict which bucket their address will fall into.

#### Incoming connections

Up to 24 incoming. The inbound cap equals the 24 outgoing cap, so the network-wide out-degree target is attainable (in any connection graph the mean in-degree equals the mean out-degree; a lower inbound cap would bound effective out-degree below 24 and weaken the eclipse bound derived for 24 outgoing). On overflow — eviction (the protected classes below sum to 20 of 24, leaving headroom to evict):

1. Protect 4 with the lowest ping
2. Protect 4 with the most recent useful messages (any valid Montana message the node has not yet seen)
3. Protect up to 8 from different subnets (one from each)
4. Protect 4 with the most recent proposals
5. From the rest — evict from the largest subnet group

#### Anchors

The 2 outgoing connections with the longest uptime are saved every τ₂. On restart after a crash or upgrade — connect to the anchors first, before any random selection from the tables.

#### Feeler

Once every 10 τ₁: connect to a random address from «new», perform the IBT handshake (all three verification levels). Success at any level → move to «verified» with the level tag (node / invited / account). Failure → mark or remove.

#### Rotation

By behaviour: if a peer has not relayed a single new proposal over τ₂ — replace it. A peer with more than 50% invalid messages in a sliding τ₁ window — disconnect, with a τ₂ reconnection ban. A peer that relays honestly is useful to the network and stays.

#### Persistent guards (Tier 2+)

For Tier 2+ deployments (see the Privacy Tier model) a node designates 2-3 of its 24 outgoing connections as a persistent guard-set, exempt from behavioural rotation, so the number of distinct peers ever used as a first hop stays small and the predecessor-attack surface does not accumulate over time. The guard-set is disjoint from the Dandelion stem_peers — a guard is never a stem successor — so long-lived-first-hop exposure and sender-deanonymization are never concentrated on the same peer. Guards remain subject to the four diversity constraints (/16, ASN, start_window). Tier 1 keeps full behavioural rotation, where 24 diverse connections already balance eclipse resistance against fast discovery of working paths.

#### PeerRecord

PeerRecord format used in peer exchange:

```
PeerRecord:
  ip_version     1B   (u8, 0x04 or 0x06)
  ip            16B   (IPv4 in the last four bytes, or full IPv6)
  port           2B   (u16 little-endian)
  node_id       32B
  start_window   8B   (u64 little-endian, from the Node Table)
= 59 bytes fixed
```

`PeerRecord` is the single peer-exchange record; the `PeerEntry` payload of `PeerListResponse` (message type 0x51) has the identical 59-byte layout. The record carries `node_id` but not the peer's ML-DSA-65 identity public key: the key is transmitted as `rs_id_pk` inside the Noise_PQ XX handshake (msg2) and authenticated there, and the connecting node verifies that `SHA-256("mt-node" || rs_id_pk)` equals the advertised `node_id`. The client's own IBT proof binds `server_node_id`, not the server's public key, so `node_id` alone is sufficient to dial and authenticate a peer. Peer exchange: at most 100 records per message; at most one peer-exchange message per τ₁ from each peer.

#### Reachability-aware ranking

When a reachability map is available for the node's own vantage (see «Reachability sensing and auto-steering» below), candidate entry points are additionally ranked by their measured reachable fraction on a supported transport profile. The random 50/50 draw between the «new» and «verified» tables, the four diversity constraints, and the absence of chain_length preference are unchanged — reachability ranking selects among candidates that already satisfy those constraints, and the local feeler probe remains authoritative over the map.

### Censorship-resistant discovery

Hard-coded discovery-seed peers `(IP, node_id, pubkey)` ship with the client so a newcomer with no peers can find the mesh; each ships as a signed entry capsule (see «Entry-point distribution» below), the bundled public key making the seed's signature verifiable before the first connection. If every seed IP is blocked at the country level — a new node cannot enter the network through that channel alone. Six independent discovery channels exist. One of the six is enough.

**1. Peer exchange.** Every node keeps and forwards a list of active peers to newcomers. Knowing the IP of a single node is enough — a friend, a QR code, a messenger. One live contact = entry to the network.

**2. DHT.** A Kademlia DHT on top of libp2p. Nodes find each other without a central point. Identifiers are randomized — the DHT does not reveal node_id before a Montana connection is established.

**3. Bridge nodes.** Nodes outside the censored jurisdiction, published through out-of-band channels (social networks, messengers, printed QR codes). The IP of a bridge node is unknown to the firewall until it is used.

**4. Encrypted Client Hello (ECH).** Bootstrap via a CDN that supports ECH. The SNI is encrypted — an observer sees the CDN IP but not the target domain. Effective in jurisdictions without active ECH-extension blocking. In jurisdictions that block ECH (China since 2023, Russia since 2024) — this channel is non-functional. For such jurisdictions — channels 1-3 and 5.

**5. Mesh peer exchange.** With no internet access at all (state shutdown, inter-zone connectivity loss, local isolation), the node discovers local peers via mesh transport (Bluetooth LE advertisement, Wi-Fi Direct service discovery). Peer exchange runs at the mesh frame level with `frame_type = 0 (discovery)` — see the «Mesh Transport» and «Store-and-Forward Semantics» subsections. The physical discovery radius is tens of metres; mesh multi-hop forwarding extends the effective radius to hundreds of metres and kilometres with sufficient device density. When at least one device in the mesh network gains internet access — the entire chain synchronizes through it as a single gateway.

**6. DNS-tunnelled bootstrap.** Where port 443 is fully blocked but recursive DNS (port 53, or DoH/DoT) remains open, a node retrieves a single live entry point through a DNS tunnel. The channel is bootstrap-only: it carries a compact peer record (`(IP, node_id)`), never session data, because DNS-tunnel throughput is small and the technique is itself fingerprintable by an attentive censor. The resolver-side responder is an external carrier of the same trust class as T2/T3 — its operational trust note is recorded in the Threat Model, and the channel degrades to channels 1-3 and 5 where DNS tunnelling is filtered.

Redundancy = resilience. The six channels are independent at the physical-delivery layer (IP internet for 1-4 and 6, radio mesh for 5). A state-level block of the internet channel does not affect channel 5 — disabling mesh requires physically suppressing Bluetooth / Wi-Fi on every device, which is practically infeasible.

#### Entry-point distribution

Cold-start entry points reach a newcomer as **entry capsules** distributed from a pool rather than as a single fixed list. A capsule is a compact, self-authenticating record of one reachable entry:

```
EntryCapsule:
  ip_version          1B    (u8, 0x04 or 0x06)
  ip                 16B    (IPv4 in the last four bytes, or full IPv6)
  port                2B    (u16 little-endian)
  node_id            32B
  valid_until_window  8B    (u64 little-endian)
  signature        3309B    ML-DSA-65 over the domain-separated preimage
= 3368 bytes fixed
```

The signed preimage is `"mt-entry-capsule" || u8(ip_version) || ip[16] || u16_LE(port) || node_id[32] || u64_LE(valid_until_window)` (fixed-width fields, little-endian integers, matching the layout above); the entry node signs it with its own ML-DSA-65 identity key. The capsule authenticates the binding of an address to a node identity and a freshness horizon, and carries no transport-profile or cover-host field — profile selection is the connecting node's local decision (Auto-steering), never a distributed instruction. Its binding KAT vector is generated by the reference implementation from the canonical preimage above, in the manner of the IBT-proof vectors.

**Invariants EntryCapsule:**
- `ip_version ∈ {0x04, 0x06}`; any other value drops the capsule.
- `ip` and `port` form the dial address; they carry no independent validity check — their authenticity rests on the `signature`, which covers them.
- `node_id` is 32 bytes and equals `SHA-256("mt-node" || rs_id_pk)` of the identity key `rs_id_pk` that verifies the signature.
- `valid_until_window ≥ known_window_index`; an expired capsule (`valid_until_window < known_window_index`) drops.
- `valid_until_window ≤ known_window_index + τ₂`; a capsule dated more than one τ₂ epoch into the future drops, bounding how far a signer can pre-date freshness.
- The `signature` verifies as ML-DSA-65 over the domain-separated preimage under the identity public key bound to `node_id` — held from the shipped discovery-seed bundle, cached from a prior contact, or obtained in the IBT handshake; a capsule whose signature does not verify under the key bound to its `node_id` drops. Because the signature covers the address and `valid_until_window`, a distributor that only relays capsules holds no signing key and can neither rebind a `node_id` to a false address nor extend its freshness.

**Pooled distribution.** Any live node, and the shipped discovery-seed bundle, distributes entry capsules; each responder returns a bounded, requester-varied subset of the pool it knows rather than the whole set. The subset is drawn by `Hash(responder_secret, requester_group)` — the address-manager bucketing rule — so two requesters from different /16 groups receive overlapping but non-identical subsets, and a single requester cannot enumerate the pool from one vantage. Distribution reuses the peer-exchange envelope and its rate limit (at most one message per τ₁ per peer, at most 100 records per message); blocking the subset one requester received neither reveals nor blocks the rest. A shipped discovery seed ships as an entry capsule plus its identity public key, so the seed's own signature is verifiable before the first connection; capsules learned from an untrusted distributor are confirmed against the key obtained in the IBT handshake.

**Scoped racing.** A connecting node races connection attempts across at most `ENTRY_RACE_WIDTH` (default 3, a bounded fan-out that masks one slow or filtered path without a connection storm) candidate capsules in parallel and adopts the first that completes the IBT handshake, so a slow or filtered candidate never stalls entry. Racing is scoped to verified non-scarce paths: candidates are drawn from those the reachability map corroborates as reachable on a baseline-eligible profile for the node's vantage (Auto-steering), and the race never fans out onto a scarce carrier — a T3 bridge or the reserve gateway-front set is tried only after the non-scarce race fails, and then singly, while the externally-observable profiles T2 and T3 stay degradation-triggered per Auto-steering rather than raced. Scoped racing therefore accelerates entry without burning the scarce reserve and without signalling tool usage while an ordinary path works.

Entry-capsule distribution, pooled subset selection, and scoped racing are local network-stack behaviour on the node's own clock and sit outside the scope of consensus state.

#### Indirect rendezvous

When no direct profile reaches any node — a path that enumerates and blocks the entire reachable node set yet leaves widely shared infrastructure usable — a newcomer and a node rendezvous asynchronously through a public whiteboard: a location on a large third-party service (a content-delivery cache path, an object-store bucket, a paste endpoint). The newcomer deposits a sealed rendezvous request — a fresh ephemeral ML-KEM public key, encrypted to a shipped discovery-seed key so the service operator stores only opaque bytes — at a location derived from a rotating index carried in the discovery-seed bundle; a node polls that location and returns an EntryCapsule, signed under `"mt-entry-capsule"` as everywhere and encrypted to the newcomer's ephemeral key, after which both parties fall back to a direct profile. The rendezvous introduces no new cryptographic primitive — it reuses the EntryCapsule signature and the ML-KEM encapsulation already in the transport suite — tunnels the inner Noise_PQ XX session unchanged, and adds no consensus state. Its reach is best-effort, not guaranteed: the rendezvous index derives from the public discovery-seed bundle, so a censor can derive and block a given location as readily as a newcomer finds it; the defence is cheap rotation over a large index space against per-location blocking — not collateral freedom — and a deposit that rides ordinary-looking content raises the per-location cost. The chosen service can also rate-limit or ban the deposit pattern, so no single service is a durable dependency; this external dependency joins the T1, T2 and T3 operational-trust class (Threat Model → transport profile external dependencies).

Indirect rendezvous is local network-stack behaviour on the node's own clock and sits outside the scope of consensus state.

### Reachability sensing and auto-steering

Filtering is applied per access network: a bootstrap address, port, or transport profile reachable from one operator is filtered on another, and the filter set shifts over time. A static entry list is therefore insufficient — a node learns which entry points are reachable from its own vantage and steers toward them. The mesh measures its own reachability and converges on working paths without operator intervention.

#### Reachability observation

A node or client that completes — or fails — a connection attempt records a reachability observation:

```
ReachabilityObservation:
  vantage_class    coarse network locator of the observer:
                   (country_code 2B ISO-3166, asn u32). No finer
                   locator is recorded; the observation does not
                   identify the individual observer.
  target_ref       32B  node_id of the attempted peer
  profile          u8   transport profile attempted (T0..T4)
  outcome          u8   0 = reachable (handshake completed),
                        1 = unreachable (timeout / reset / filtered)
  observed_window  u32  cached window_index at observation time
```

Observations are transport-layer telemetry on the node's local clock. They enter no state root and form no consensus state. They are bounded and ephemeral: a node retains at most `MAX_OBSERVATIONS_PER_VANTAGE = 256` per `(country_code, asn)` pair — mirroring the `used_online_nonces` per-key bound — and prunes any observation older than `7 × τ₁`, the mesh-IBT staleness bound, so the working set is memory-bounded independently of network size.

#### Reachability map

A node aggregates observations into an advisory map `(vantage_class, target_ref, profile) → reachable_fraction`, where `reachable_fraction` is the count of `outcome = 0` over total observations for that triple. The map propagates as `ReachabilityAdvert` records over peer exchange, under the existing peer-exchange rate limit (at most one peer-exchange message per `τ₁` per peer, at most 100 records per message). The map is advisory: it ranks candidate entry points and never authorizes a connection by itself.

```
ReachabilityAdvert:
  country_code      2B   ISO-3166-1 alpha-2 of the vantage
  asn               4B   u32, autonomous system of the vantage
  target_ref       32B   node_id of the observed peer
  profile           1B   transport profile observed (T0..T4)
  reachable_num    u16   corroborating observations with outcome = 0
  reachable_den    u16   total observations for the triple
  observed_window  u32   cached window_index of the latest observation
```

**Invariants ReachabilityAdvert:**
- `country_code` is two ASCII letters in the ISO-3166-1 alpha-2 set; any other value drops the record.
- `profile ∈ {0, 1, 2, 3, 4}` (T0..T4); any other value drops the record.
- `reachable_den ≥ 1` and `reachable_num ≤ reachable_den`; otherwise the record drops. `reachable_fraction = reachable_num / reachable_den` is a ranking ratio only and forms no consensus state.
- `observed_window` lies within `[known_window_index − 7 × τ₁, known_window_index]`; a staler value drops the record.
- A `(country_code, asn, target_ref, profile)` triple is acted on only when corroborated by records from at least `REACHABILITY_QUORUM = 3` distinct /16 source groups within that `(country_code, asn)` vantage; the /16 is the diversity unit of the outgoing-connection constraints. A single advert ranks candidates and authorizes no connection.

#### Auto-steering

When selecting an entry point a node consults the map for its own `vantage_class`, prefers candidates with the highest `reachable_fraction` on a supported transport profile, and breaks ties by the round-trip latency of a feeler probe. The node then confirms the chosen candidate by its own connection attempt and IBT handshake; the map informs the choice and the local probe is authoritative, mirroring the feeler discipline of peer selection. On loss of a working entry mid-session the node re-steers to the next corroborated candidate; one active entry carries a hot reserve. To keep alternatives warm ahead of a block rather than only after one, a node continuously directs a minimum `REACHABILITY_EXPLORE_FRACTION` (default 0.1) of its feeler probes to profiles below the current best. Exploration on externally-observable profiles (T2, T3) is triggered only by observed degradation of the current entry and never enumerates the reserve bridge set, so it neither burns scarce bridge addresses nor signals tool usage while the base profile works. When several corroborated candidates are available the node races them under the scoped-racing discipline (Censorship-resistant discovery → Entry-point distribution), drawing only from map-verified non-scarce paths and leaving a scarce carrier for a single fallback attempt.

The baseline profile is vantage-dependent. In a vantage whose access network is known to fingerprint or filter Montana traffic — identified by the node's own `(country_code, asn)` against a shipped hostile-vantage set, or by the reachability map showing T0 fingerprinted or throttled — the node selects a genuine-carrier profile (T1 or higher) as its baseline rather than T0, so the bare Noise_PQ handshake is never exposed to a DPI adversary. In a benign vantage T0 remains the default for lowest latency.

#### Quorum and diversity

A single reported observation does not move the map: a `(vantage_class, target_ref, profile)` triple is treated as reachable only when corroborated by observations from at least `REACHABILITY_QUORUM = 3` distinct /16 source groups — the diversity unit of the outgoing-connection constraints. An adversary reporting false reachability from one position therefore cannot steer honest nodes toward a dead or hostile entry: honest nodes require independent corroboration across network positions and ultimately verify by direct IBT probe.

Reachability sensing, the map, and steering are local network-stack behaviour on the node's own clock and sit outside the scope of consensus state.

#### Gateway entry-point failover

Where clients reach the network through a managed gateway rather than by direct peer connection, the same reachability discipline governs the gateway tier. A client holds a single stable entry identifier; the network binds that identifier to the currently reachable gateway and rebinds it on failure, so the client never reconfigures.

An ordered set of gateway fronts backs the identifier. Election is deterministic: the highest-priority front whose transport handshake currently succeeds owns the identifier. Liveness is decided by a completed transport handshake of the deployed obfuscation profile, not by a bare TCP or ICMP probe — a port that accepts a connection but does not complete the handshake is not a live gateway, so an unrelated service answering on the same port is never mistaken for a working front. A front is demoted only after a corroborated run of failed handshakes (hysteresis) and reclaims its priority on recovery; rebinding moves the same identifier between fronts that present identical client-facing transport parameters, so sessions migrate with no client-visible change.

External exposure is minimized as a censorship countermeasure. Only the currently bound entry is externally resolvable; reserve fronts and exit relays are not enumerable from outside and appear at the entry identifier only when promoted. An adversary who resolves the identifier learns exactly one address — blocking it triggers rebinding to a reserve for which the adversary holds no prior address, rather than collapsing a pre-enumerated fleet. Exit relays are reached only through a front and are never client-facing, so the client-visible surface is one address at a time regardless of fleet size.

By preference the bound entry is a single address reachable inside the client's own jurisdiction: a domestic first hop minimizes the connection signature and relocates the cross-border segment to the gateway-to-exit link, which carries datacenter-grade capacity under the same handshake obfuscation. The gateway tier is deployment infrastructure: it enters no state root and forms no consensus state.


### Dandelion++ (sender anonymity)

Montana's P2P gossip retransmits operations through all nodes. Without protection, the first peer knows the sender's IP. Dandelion++ (Fanti et al. 2018) breaks the IP → operation link by modifying the existing gossip.

**Two phases:**

```
Stem:
  The operation traverses a chain of random nodes (2-3 hops on average).
  Each node sees only the previous hop, not the originator.
  At each hop, with probability p = 0.4, the operation transitions to fluff.
  E[stem_length] = 1/p = 2.5 hops.
  P(stem ≤ 1) = 40%, P(stem ≤ 3) = 78%.

Fluff:
  The last stem node initiates a normal gossip broadcast.
  To the rest of the network the operation «appeared» from a random point.
```

**Stem routing.** The stem uses only outgoing connections — incoming connections do not participate. Every 693 windows the node re-selects 2 of its 24 outgoing as `stem_peers` (the stem-set selection period). Inside this 693-window window, the `stem_successor` (forward choice between the 2) rotates every τ₁. This section is the normative source for the stem parameters; the Dandelion++ closure card references it. All stem operations in the epoch are routed through one of these 2 (chosen by hash(msg)).

**Application per object type:**

| Object | Mode | Reason |
|--------|-------|---------|
| UserObject (Transfer Mode A/B, Anchor, ChangeKey, CloseAccount) | Stem → fluff | Hide sender IP |
| ControlObject (NodeRegistration) | Stem → fluff | Hide the registering candidate's IP |
| SSHA Reveal | Direct gossip (no stem) | node_id is public in the reveal, anonymity is impossible; IP is hidden by Transport Obfuscation (Noise_PQ XX over TCP/8444 with uniform framing) |
| Confirmation | Stem → fluff | Hide which node confirmed first |

**Properties:**

| Threat | Defence |
|--------|--------|
| Peer sees sender IP | Stem: the peer sees only the previous hop |
| Global observer (ISP) | Noise_PQ XX + uniform framing (Transport Obfuscation) |
| Gossip-graph analysis | The operation enters gossip from a random point |
| Control of k nodes | Deanonymization requires control of O(√n) nodes |

**Implementation:**

```
stem_peers = random_sample(outbound, 2)    // every 693 windows (selection from 24 outbound)

on_receive_stem(msg, from_peer):
  if random() < 0.4:
    gossip_broadcast(msg)                  // fluff
  else:
    next = stem_peers[hash(msg) % 2]      // deterministic choice between the 2
    send_stem(msg, next)                   // continue stem
  start_timer(msg, τ₁/2)                  // safety timer at each hop

on_timer_expired(msg):
  if msg not observed in gossip:
    gossip_broadcast(msg)                  // forced fluff
```

Each stem node insures the next. The τ₁/2 timer runs independently at every hop. If the next hop dropped the message — the current hop notices the operation's absence in gossip and performs fluff itself. Maximum latency = τ₁/2 (one hop), not cumulative.

Dandelion++ requires no external infrastructure. Every Montana node already is a relay — gossip exists; stem adds 2-3 hops in front of it. Latency overhead: milliseconds.

### NAT Traversal

A personal network works when everyone can join. Most home users sit behind NAT — invisible to incoming connections. Without NAT traversal a personal internet = a server club.

Three mechanisms, each next one used if the previous one fails:

**1. AutoNAT (detection).** The node asks outbound peers: «do you see my IP:port directly?» If yes — no NAT. If no — the node knows its NAT status.

**2. DCUtR (hole punching).** Two NAT'd nodes coordinate through a third node with a public IP. Both send outgoing packets — the routers open «holes» for the replies. After coordination — direct connection. Success: 60-70% of cases (TCP). Carrier-grade NAT (mobile operators): ~30%.

**3. Circuit Relay v2 (transit).** If hole punching fails — traffic flows through an outbound peer with a public IP. Relay is not a separate mechanism and not a dedicated server. A relay connection = an ordinary outgoing connection, subject to the same rules: uniform framing, diversity constraints, behavioural rotation. The content is end-to-end encrypted (Noise) — the relay sees participants' IPs but not the content. Metadata is spread across 24 outbound peers from different /16s and ASNs — no single relay sees the full graph.

Relay is not a fallback but a connectivity guarantee for any NAT type. Hole punching is an optimization to reduce relay load.

**Relay limits:** up to 32 concurrent relay connections per node, bandwidth per relay ≤ baseline frame rate (1 KB/s). 32 × 1 KB/s = 32 KB/s ≈ 82 GB/month — acceptable for a home node with a public IP.

**Obligation.** Nodes with public IPs support relay — a personal network works when everyone can join. The reference implementation enables relay when a public IP is detected. Feeler connections check relay support on peers; nodes without relay are marked `no-relay` in the address manager. NAT'd nodes prefer relay-capable peers when picking outgoing connections.

#### Volunteer transit-relay role

A node in an uncensored jurisdiction may operate as a volunteer transit-relay without completing the sequential-SHA-256 node-registration barrier. A transit-relay forwards opaque Noise_PQ XX ciphertext (Circuit Relay v2) and nothing else: it never gossips, never enters peer tables as a node, never serves as a Dandelion stem hop, and holds no consensus role. It carries no consensus state and enters no state root.

A relay observes both endpoints' IP addresses (never their content, which is end-to-end encrypted). An unbounded barrier-free relay pool would therefore let one actor cheaply harvest a who-talks-to-whom graph at the IP layer — a Sybil on metadata rather than on consensus. This is bounded by time, not by money or registration, per [I-15]:

- A relay identity is eligible to carry transit for a served node only after it has been continuously reachable, as observed by that node, for `RELAY_SENIORITY_WINDOWS` (operator-configurable local default, 336 windows — one selection interval). A fresh identity cannot immediately become a transit path, so a mass pool of throwaway relays confers no immediate advantage.
- A node distributes its relayed connections under the same /16 and ASN diversity constraints as its outgoing connections, so no single actor's relays dominate a node's transit set.
- Relay bandwidth per relay ≤ the baseline frame rate (Relay limits above).

Seniority-through-continuous-reachability is a time-market barrier native to the protocol: it costs an honest volunteer nothing beyond staying online, yet a censor cannot buy seniority, so a mass relay pool is expensive in the only resource Montana meters — time. The lifecycle of a relay identity is purely local and ephemeral: a served node drops it from its transit set the moment continuous reachability lapses, and re-accrual restarts the seniority clock.

All three NAT-traversal mechanisms are libp2p standards (AutoNAT, DCUtR, Circuit Relay v2). Zero new protocol primitives; the volunteer transit-relay is a time-gated policy over Circuit Relay v2, not a new object.

#### Metadata-private mailbox access (federated clients)

A federated client — an account holder that runs no node of its own — reaches its blind message roles over volunteer transit-relays rather than by a direct connection, and holds its two identifying markers — its account and its conversations — apart, so that no single node holds both. The roles are served by ordinary registered nodes; no separate relay infrastructure exists. Each client↔node leg carries an `access_class`: `direct` (an IP connection) or `indirect` (opaque Noise_PQ ciphertext forwarded through a volunteer transit-relay, see «Volunteer transit-relay role»). A federated client uses `indirect` legs only; a sovereign client (own node) needs none of this — its relay is local and has no third party.

**Federation and authentication.** The federation is the registered node set (the Node Table), gated by the sequential-SHA-256 admission barrier; no new membership mechanism is introduced. A native federated client (iOS / Android / desktop) is a light client: it obtains an authenticated state root by verifying the signed cemented header chain — each cemented proposal header is signed by its confirmer set in ML-DSA-65 — anchored at the genesis manifest, never taken on any node's word. It then authenticates any node against that root by two independent checks: the IBT handshake binds the node's ML-DSA-65 key to its `node_id`, and a Merkle inclusion proof shows that `node_id` is in the Node Table; placement is cross-checked across the committee members below, so a single lying node cannot misdirect. The manifest pins only the bootstrap relays used at cold start, before the header chain is synced — the role the discovery seeds play. A web-only client that cannot verify the header chain runs in a **reduced-assurance mode**: it trusts the manifest bootstrap relays for the state root, keeping the stored-label unlinkability and the transit IP-hiding but not the full node-verified anti-targeting guarantee.

**Inbox committee and session node.** Two roles serve a federated client, held apart because they derive from different material:

- the **inbox** — the account marker, holding the stable account-derived inbox label for first contact and the wake signal — derives from the recipient's **public** `account_id`, so a fixed placement would be targetable. It is therefore not a single relay but an object in the erasure-coded data substrate (Montana Messenger, backup substrate): stored `m`-of-`k` across the `k` nodes closest in the Node-Table keyspace to `H(account_id ‖ epoch_seed)`, where `epoch_seed` (below) makes the placement unpredictable and reshuffles it each selection interval. A sender reaches it from the public `account_id` and the public `epoch_seed`. Durability, proof-of-storage, and re-replication as the committee reshuffles are the substrate's existing machinery, so first contacts persist for the full retention window across rotation — the rotation that resists targeting and the persistence that first contact needs are both carried by the substrate, not traded against each other.
- the **session node** — the conversation markers, holding the rotating session labels — derives from the two conversants' **shared secret** `routing_secret`, so its placement `H(routing_secret)` is unpredictable to any observer without the secret and needs no seed rotation to resist targeting. It is a single node and stays fixed for the session, so session messages persist on it; on the rare event that the node leaves the network the conversants re-derive to the next node and resend the gap. Only the public-derived inbox rotates; the secret-derived session node does not.

`epoch_seed` is the network-bound unpredictable seed carried forward from the previous selection interval — the cemented aggregate finalized at the current interval's boundary, the seed class that drives the node lottery per [I-8]. It is available to every honest participant throughout the current interval and was unpredictable until the previous interval cemented, so inbox placement is computable when needed yet cannot be pre-targeted: an adversary cannot position a `node_id` near a future `H(account_id ‖ epoch_seed)`, and admitting a node near a chosen keyspace point costs a full `τ₂` admission epoch — far longer than a selection interval — so the seed rotates before the adversary can arrive.

**What each role sees.** The session node sees the rotating session labels but never the account marker; put and poll of session labels are split across different transit-relays (no single transit carries two of one client's legs in a `τ₁` window, under the /16 and ASN diversity constraints), so the session node observes each leg from a different transit address, never the client's, and learns at most one anonymous session's label activity over time — volume and timing, no account. Each inbox-committee node sees that the account marker is stored there — the fact of the account's activity — but the first-contact content is sealed (sealed-sender) and requires `m`-of-`k` committee shards to reconstruct. **IP linkage** — binding a physical address to a conversation — requires colluding two transit-relays with the session node (≥2 /16). **The account who-graph** — binding `account_id` to its conversations — requires colluding the inbox committee with the session node: neither role alone links an account to a session label. The residuals are honest and bounded: an adversary holding a fraction `f` of nodes sits on about `f` of inbox committees per interval and reconstructs first-contact content only where it holds `m`-of-`k` of a committee — the honest-majority bound, reshuffling each interval, not a per-account targeting primitive; the cross-role timing signal (the committee wakes an account, the session node later sees a poll) is available only to the committee and the session node in collusion, and canonical polling **reduces but does not eliminate** it. Clients poll the inbox and the session labels on the shared canonical `τ₁`-window schedule — jittered within the window by an activity-independent distribution, not reactively on message arrival — which decouples the fine-grained timing (the wake-to-immediate-poll coincidence) and makes each window's polls a synchronized herd. It does not anonymize the inbox side, however: the committee holds the account-derived inbox label and so observes account `B`'s own inbox and wake events, and over many such events the committee and the session node in collusion can still correlate `B`'s coarse presence pattern with a session's activity — weakened by the sparseness of inbox and wake events and by the canonical schedule, but not removed. Full closure — making the observable poll pattern independent of the user's real presence — requires an always-on poller, i.e. the sovereign node (the low-latency persistent-push mode is the opposite trade, revealing presence through the continuously-open socket). Federated timing is therefore reduced, not eliminated; the sovereign node remains the full closure, in line with the network's general stance on timing correlation (see «Canonical-aggregation closure for timing-correlation»). A degenerate deployment that serves both roles from one node collapses the account who-graph to that node; full closure with no third party at all is the sovereign node. A private-information-retrieval or MPC inbox would hide the account-presence signal from the committee nodes, but per-query private retrieval over the entire account keyspace is infeasible on commodity hardware at network scale; the full closure of the presence signal is the sovereign node.

Metadata-private mailbox access, role placement, and light-client verification are local network-stack behaviour on the node's own clock; the epoch seed, Node-Table membership, and header chain are read from consensus state, and this section adds no consensus state of its own.

### Mesh Transport

Internet is not always available. State shutdowns (Iran 2019 — a week, Belarus 2020 — days, Myanmar 2021 — months), local outages, isolated zones. Montana keeps working under these conditions through mesh transport over Bluetooth Low Energy and Wi-Fi Direct — devices discover each other in physical radius and forward encrypted Montana messages hop by hop. Mesh does not replace internet transport, it complements it: when connectivity returns, the network automatically converges through a mesh-internet gateway.

**Mesh transport is orthogonal to consensus**, just like internet transport ([the section above](#network-layer)) — the state machine works over any delivery channel without changes.

#### MeshFrame wire format

All mesh messages are fragmented into fixed-format frames:

```
MeshFrame:
  mesh_protocol_version   u16 LE — mesh wire format version
                                   (0x0001 for v1)
  frame_type              u8     — 0=discovery, 1=data,
                                   2=ack, 3=forward
  ttl                     u8     — max 16 on creation, monotonic
                                   decrement; decremented at each
                                   forwarding hop; ttl=0 → drop
  hop_count               u8     — 0 on creation, monotonic increment;
                                   incremented at each forwarding hop
  sender_ref              32B    — mesh_session_id of the initiator
                                   (not the direct node_id, for
                                    privacy at the mesh layer)
  recipient_hint          32B    — encrypted routing hint, or the
                                   broadcast marker (0xFF × 32)
  payload_length          u16 LE — payload length in bytes, ≤ 256
  payload                 variable — encrypted blob, ≤ 256B to fit one
                                     BLE GATT notification without
                                     fragmentation
  mac                     16B    — HMAC-SHA-256 truncated to 16 bytes,
                                   recomputed by each transmitting hop
                                   under the pairwise link session key
                                   (link-local; see invariants)

Total: 71B header + 256B payload + 16B MAC = 343B max
       (header = 2+1+1+1+32+32+2)
```

**MeshFrame invariants:**
- all multi-byte integer fields (`mesh_protocol_version`, `payload_length`) are little-endian, consistent with the ProtocolMessage envelope
- `mesh_protocol_version` ∈ {0x0001} for v1; any other value → reject
- `frame_type` ∈ {0, 1, 2, 3}; any other value → drop
- `ttl` ∈ [0, 16]; the sender sets it ≤ 16 on creation; at each forward `ttl := ttl - 1`; if `ttl = 0` and the frame requires forwarding → drop
- `hop_count` ∈ [0, 16]; 0 on creation; at each forward `hop_count := hop_count + 1`; if `hop_count > 16` → drop (guard against a malformed increment)
- `sender_ref` = 32 bytes = the initiator's mesh_session_id (see derivation below), not a direct node_id
- `recipient_hint` = 32 bytes; the value `0xFF × 32` marks a broadcast, any other value is an encrypted routing hint; the receiver checks a match against its own local state
- `payload_length` ≤ 256; a larger value → drop
- `payload` is exactly `payload_length` bytes; an encrypted blob (encryption happens at the session layer — the mesh transport sees only ciphertext)
- `mac` = 16 bytes = HMAC-SHA-256(`link_mac_key`, header_bytes || payload) truncated to the first 16 bytes, where `link_mac_key` = HKDF-SHA-256(`session_shared_secret`, salt = empty, info = "mt-mesh-frame-mac", length = 32) of the pairwise session between the transmitter and the receiver of **this** hop. The MAC is **link-local**: it authenticates one radio hop, not the end-to-end path. A forwarding node verifies the incoming MAC under its session with the previous hop, then recomputes the MAC under its session with the next hop before retransmitting — so covering the mutated `ttl` / `hop_count` is correct, because every hop signs the exact bytes it transmits. A mismatch → drop + increment the soft-blacklist counter for the peer this frame arrived from.
- End-to-end authenticity of the payload does **not** rest on the link MAC. The payload is end-to-end encrypted and authenticated to the final recipient at the session layer (ML-KEM-768 session key), independent of any intermediate hop; the link MAC provides only immediate-link integrity and soft-blacklist accounting. A MeshFrame carries no ML-DSA-65 signature of its own — identity-level authentication happens once, at the mesh IBT handshake that established each pairwise session (formula `mesh_proof`), and subsequent frames on that link are authenticated by the link MAC.
- Cross-field consistency: `hop_count + ttl = 16` at all times (initial: hop_count=0, ttl≤16; each forward decrements ttl and increments hop_count, so the sum is invariant); `hop_count + ttl ≠ 16` is a malformed frame → drop + increment the soft-blacklist counter for the peer this frame arrived from

**mesh_session_id derivation.** For each mesh session (between a pair of peers after the mesh IBT handshake):

```
mesh_session_id = HKDF-SHA-256(
    ikm    = shared_secret_from_noise_handshake,
    salt   = mesh_session_nonce_initiator || mesh_session_nonce_responder,
    info   = "mt-mesh-session",
    length = 32
)
```

`mesh_session_id` occupies the `sender_ref` field instead of a direct `node_id`, so the mesh wire format does not reveal the sender's identity to a random listener in radio range. The identity is revealed only to a peer with which a session is established (they both know the mesh_session_id).

**MeshFrame validation.**

1. `mesh_protocol_version` matches the peer's expected version. Mismatch → drop, no forward.
2. `frame_type ∈ {0, 1, 2, 3}`. Otherwise → drop.
3. `ttl ∈ [0, 16]`. If ttl=0 and the frame arrived for forwarding → drop.
4. `hop_count ≤ 16`. Otherwise → drop (guard against a malformed increment).
5. `payload_length ≤ 256`. Otherwise → drop.
6. `mac` verified via HMAC-SHA-256 under the link session key of the hop this frame arrived on. Mismatch → drop, increment the soft-blacklist counter for the arrival peer.
7. For `frame_type = 3 (forward)` — apply the Store-and-Forward Semantics rules (below), recomputing the link MAC for each onward hop.

#### Mesh framing profile

The internet-transport uniform framing (the «Uniform Framing» subsection) does not apply to mesh transport. Mesh has an independent framing profile:

```
Internet (existing):
  frame_size            = 1024 bytes
  baseline_rate         = 1 frame/s
  context               = Noise_PQ XX AEAD stream over TCP/8444

Mesh (v1):
  frame_size            = 256 bytes (fits a typical BLE MTU without
                          application-level fragmentation on most
                          iOS / Android stacks)
  baseline_rate         = 1 frame / 10 s (baseline advertisement plus
                          occasional data, battery-sustainable)
  burst_mode_rate       = 1 frame/s (activated ONLY during an active
                          mesh chat session, not continuous)
  burst_mode_duration   = ≤ 120 s after the last data frame, then
                          return to baseline_rate
  fragmentation         = sequence numbers for messages > 256B,
                          reassembled at the recipient via seq_id in
                          an application-level payload header
```

Parameter rationale:
- 256B — the BLE MTU varies in practice from 23 to 512B; most modern iOS / Android devices support ≥ 247B, and 256B is chosen as a fit-without-fragmentation compromise on mainstream devices.
- 1 frame / 10 s baseline — continuous Bluetooth scanning at a faster rhythm consumes 30-50% of a phone battery within a few hours; the 1/10s profile extends battery usability to a working day.
- burst up to 1/s — active messaging requires reasonable responsiveness; activation on the «active chat session» event confines the energy spike to real use.

#### Fragmentation

Messages larger than 256B are fragmented at the application level before being enqueued into the mesh:

```
ApplicationPayload (before fragmentation):
  fragment_count     u16   — total number of fragments
  fragment_index     u16   — index of the current fragment (0-based)
  message_id         32B   — unique message id, shared across all
                             fragments
  data               variable — a part of the encrypted payload
```

The recipient reassembles fragments by `message_id`, with order restored through `fragment_index`. Reassembly timeout: τ₁ from the first fragment received (on the transport stack's local clock, outside the consensus-time scope) — if not all fragments arrive, the partial is dropped. `fragment_index ≤ 255` (max 256 fragments × 256B payload = 64KB upper bound for one application message; larger volumes go through Content-Layer chunking at the blob level).

#### Mesh discovery flow

1. A device in mesh-active mode periodically (baseline_rate = 1 frame / 10 s) broadcasts `frame_type = 0 (discovery)` with `sender_ref = mesh_session_id_self_generated` and `payload` = a short advertisement: protocol version, capability flags, optional trust hint.
2. Other devices in range receive the broadcast and extract the advertisement.
3. If the receiving device considers the initiator potentially interesting (a known contact in the address book; a broadcast addressed to the broadcast marker while the device is in broadcast-listening mode; any other application rule) it initiates the mesh IBT handshake (see «Identity-Bound Tunnel» above, the `mesh_proof` formula).
4. After a successful handshake the session is established and both peers add `mesh_session_id` to their active sessions.
5. Data is exchanged through `frame_type = 1 (data)`.

#### Battery management

The reference implementation recommends:
- Scheduled Bluetooth scan: once every 10 seconds at baseline, more often during a burst.
- Wi-Fi Direct is used only for high-throughput sessions (large file transfer), not continuously.
- iOS background-mode constraints: full mesh transport runs only in the foreground; in the background, limited Core Bluetooth BGTaskScheduler scanning is available.
- Android: BLE advertisement and scanning in the background are a platform standard, requiring a declared foreground-service notification for compliance.

### Store-and-Forward Semantics

Mesh transport is inherently asynchronous: the recipient of a message may be out of range at the moment of sending. Store-and-forward semantics describe how intermediate devices buffer and forward messages to their final recipient.

#### Buffer model

Every device in mesh-active mode maintains a local buffer:

```
MeshBuffer (device-local state):
  entries: map<frame_hash, BufferEntry>

BufferEntry:
  frame              MeshFrame
  receipt_seq        u64       (local sequence counter, incremented on
                                each receive; used for local FIFO
                                ordering in the buffer; not sent over
                                the network)
  ttl_remaining      u8        (decremented each forwarding hop)
  sender_ref         32B       (from the frame, for the per-sender quota)
  forwarded_to       set<peer_id>  (peers already forwarded to,
                                    a loop guard)
```

`frame_hash = SHA-256(MeshFrame serialized)` is the key for idempotent receipt.

#### Buffer policies

**Capacity limits (defaults, tunable in the implementation):**
- Max buffer size per device: 1024 frames (≈ 336 KB).
- Max retention per frame: 1440 τ₁ (TTL expiry on the buffer entry, emergently ≈ one day at genesis calibration; independent of the `ttl` in the frame, which is decremented per hop).
- Max frames per sender_ref in the buffer: 10 concurrent.

**Priority queue (on enqueue):**
1. Own sent frames (originated by this device) — highest priority.
2. Frames addressed to known contacts (stored locally) — high priority.
3. Frames addressed to unknown recipients (broadcast or an unknown recipient_hint) — low priority.

**Drop policy on overflow:**
- First on overflow — drop the oldest low-priority frame.
- When low-priority is exhausted — drop the oldest high-priority frame (never own).
- Own frames are not dropped before expiry.

#### Per-sender quota

Protection against flood DoS (vector M1 from adversarial review):

```
Rate limits per sender_ref:
  max_frames_per_τ₁      = 10
  max_frames_in_buffer   = 10 concurrently

On exceedance:
  - New frames from this sender_ref are dropped.
  - The sender_ref receives a signed rate-limit ack with a refusal.
  - Local soft-blacklist: exponential backoff in τ₁ — first
    violation 1 τ₁ ignore, second 2 τ₁, and so on up to a
    maximum of 60 τ₁.
```

#### Signed rate-limit acks

The relay signs an acknowledgement for each frame it accepts (and forwards or stores):

```
MeshAck:
  acked_frame_hash    32B   — SHA-256 of the frame being acked
  relay_node_id       32B
  status              u8    (0=accepted, 1=buffered,
                             2=forwarded, 3=rejected_quota,
                             4=rejected_expired)
  ack_seq             u64   — relay-local monotonically increasing
                              ack sequence counter; carries no time,
                              only the order in which this relay
                              issued its acks
  signature          3309B  — ML-DSA-65_sign(
                                relay_privkey,
                                "mt-mesh-ack"
                                || acked_frame_hash
                                || relay_node_id
                                || status
                                || ack_seq)
```

**MeshAck invariants:**
- `acked_frame_hash` = SHA-256 over the canonical serialization of the MeshFrame the ack refers to; the ack receiver verifies the hash matches a frame actually sent.
- `relay_node_id` = SHA-256("mt-node" || relay_pubkey); the receiver must know `relay_pubkey` to verify the signature.
- `status ∈ {0, 1, 2, 3, 4}`; a value outside the range → reject the ack as malformed.
- `ack_seq` — a relay-local u64 sequence counter, incremented by 1 for each ack this relay issues; used only for local ordering of acks at the receiver; carries no temporal semantics (neither wall-clock nor duration); not consensus-critical and not part of any state transition.
- `signature` = 3309 bytes ML-DSA-65, verified with `relay_pubkey`; the signed message is canonically serialized in field order (acked_frame_hash || relay_node_id || status || ack_seq).
- Signature verify rule: ML-DSA-65.verify(relay_pubkey, domain_separator || canonical_payload, signature) = valid; otherwise drop the ack.
- An ack applies to no state transition — it is purely a local signal for sender rate adjustment, outside the consensus scope.

The sender uses the ack to:
- Confirm the frame was accepted (status ∈ {0, 1, 2}).
- Detect overload (status=3 → flood suppression, reduce rate).
- Detect expired frames (status=4 → frame outdated, do not retry).

The absence of an ack within τ₁/2 of sending → the sender assumes the relay is unavailable and tries another peer.

#### Forwarding algorithm

```
on_receive(frame, from_peer):
  frame_hash = SHA-256(frame)
  if frame_hash in buffer:
    return  # duplicate, already processed

  if not validate_frame(frame):
    drop; increment soft-blacklist counter from_peer
    return

  if frame.sender_ref in soft_blacklist:
    drop silently
    return

  if buffer.sender_count(frame.sender_ref) >= 10:
    send_ack(frame, status=3)  # rejected_quota
    return

  if frame.recipient_hint matches self:
    deliver_to_application(frame)
    send_ack(frame, status=0)
    return

  if frame.ttl == 0:
    drop; send_ack(frame, status=4)
    return

  # forward case
  frame.ttl -= 1
  frame.hop_count += 1

  buffer.add(frame)
  send_ack(from_peer, frame, status=1)  # buffered
  recompute link MAC for the next hop  # link-local MAC per hop

  # opportunistic forwarding
  for peer in active_mesh_peers:
    if peer not in frame.forwarded_to and
       peer != from_peer and
       peer accepts forwarding:
      send(peer, frame)
      frame.forwarded_to.add(peer)
      send_ack(from_peer, frame, status=2)  # forwarded

on_timer_expired(entry):
  # local buffer expiry, independent of frame.ttl
  buffer.remove(entry)
```

#### Interaction with the internet

When a device gains internet connectivity it optionally (per user setting) forwards buffered mesh frames into the internet network:

1. For each frame in the buffer whose `recipient_hint` can be resolved to an account_id.
2. Forwarding via ordinary P2P gossip to the recipient's Account Host.
3. After a successful acknowledgement from the internet side, the frame is removed from the mesh buffer.
4. The internet-to-mesh direction is symmetric: a device with internet receives a message for an offline recipient and enqueues it into the mesh buffer for forwarding through nearby peers.

This makes an internet-connected device a **gateway** between the internet network and an isolated mesh region. A single such gateway restores connectivity for an entire mesh cluster to the outside world.

### Privacy Scope

Before describing the seven layers of network defence, this section fixes the **precise boundaries** of what Montana protects at the network layer and what is **deliberately not closed**.

#### Three-level decomposition

Privacy in Montana operates on three separate levels, each with its own guarantees:

1. **Wire-level (transport layer):** what a provider / DPI / single-link observer sees.
   - Defence: Noise_PQ XX (ML-KEM-768 KEM + ML-DSA-65 identity + ChaCha20-Poly1305 AEAD) + IBT + Uniform Framing + Transport Randomness + Dandelion++ + Label Rotation per τ₁ + censorship-resistant discovery.
   - Closes: local DPI, ISP, a regulator intercepting one link, small-to-medium Sybil eclipse, long-term recipient linkability through application providers.
   - **Does not close:** a global passive adversary with GPS-precision timing-correlation on ALL backbone links simultaneously (an open research problem of the whole anonymity-network field).

2. **Content-level (application data):** what a hosting node or a proposal-broadcast observer sees.
   - Defence: Anchor + ML-KEM-768 encryption (a data_hash in the proposal, content encrypted off-chain under the recipient key) + PQ Double Ratchet for messenger end-to-end + a PQXDH asynchronous handshake.
   - Closes: content surveillance by the host, recipient identity through ephemeral labels.
   - **Does not close:** endpoint compromise (RAT / spyware) — out of scope for any network protocol.

3. **Financial-level:** what any observer of a cemented proposal sees.
   - Status: **open per [I-2]** — balances, sender_account_id, recipient, amount are public.
   - This is an **intentional design feature** for regulatory compatibility (FATF / MiCA), auditability and [I-3] determinism of consensus state.
   - Privacy of financial movements at the protocol level is **deliberately not provided**.

#### Privacy Tier model for users

A user selects a privacy level by trading off against latency and bandwidth:

| Tier | Stack | Latency | Bandwidth | Threat closure |
|------|-------|---------|-----------|----------------|
| **1 default** | Seven layers baseline (Noise_PQ XX + IBT + Dandelion++ + Label Rotation + Mesh) | <500ms | minimal | Local DPI, ISP, a regulator (one link), small-to-medium Sybil, long-term recipient unlinkability, quantum store-now-decrypt-later at the transport |
| **2 recommended** | + own node (Light-Node-at-Home) + Tor entry | 500ms — 2s | medium | + the ISP does not see «Montana traffic», + bypass of a legal request |
| **3 paranoid financial** | + canonical mempool buffering + end-of-window batch flush + 100% cover traffic | 60-120s | full | + temporal unlinkability of the sender's local act → the operation's appearance in state |
| **4 research-grade** | + an artificial random delay of 5-30 min + multiple onion paths | 5-30 min | full | + extreme protection from mass-surveillance flow tracking; **practically maximum** under non-information-theoretic anonymity |

Tier 1 is automatic for all users. Tiers 2-4 are opt-in.

#### Canonical-aggregation closure for timing-correlation (a unique Montana property)

Montana differs from Tor / Loopix / I2P **fundamentally** in how operations are handled at the network layer:

```
Tor / Loopix / I2P model:
  packet_α emitted immediately → traverses the network → arrives at recipient
  Adversary observes individual packet timing → correlates Alice IP ↔ Bob IP
  Deanonymization after 100-1000 messages

Montana model:
  Alice creates operation_α locally at time T_local
  → mempool buffer on the operator node (random hold of N windows)
  → Dandelion++ 2-hop stem forwarding
  → operator's mempool → operator selection lottery
  → aggregation: proposal_W = canonical(op_α, op_β, ..., op_ω)
                 includes 100-10000 operations from different accounts
  → broadcast the aggregate
  Adversary observes the aggregate proposal; individual op_α is
  indistinguishable from op_β
  Deanonymization requires 10⁶-10⁸ messages (orders of magnitude harder)
```

**The unique Montana primitives** that work together for this property:

- **Mempool temporal buffering** — Alice's individual emit timing is decoupled from the operation's appearance in state.
- **Canonical proposal aggregation** — a proposal contains operations from many accounts, mixing the individual flow.
- **Sequential-chain canonical aggregation point** — all proposals are published at canonical moments (end-of-window), a homogeneous stream of events for the adversary.
- **Block-lattice independent AccountChain** — each account writes to its own chain, structurally separate but bundled in the proposal.
- **Dandelion++ stem buffering** — the operator entry is hidden through 2-hop stem propagation.

Operations in Montana are **state events on the canonical TimeChain**, not ephemeral network messages. This fundamental architectural property is absent from Tor (per-message routing), Loopix (per-message Poisson mixing) and I2P (per-tunnel routing). It gives Montana a **uniquely strong** weakening of global passive timing-correlation — **not an absolute closure** (the open research problem remains), but **orders of magnitude stronger** than existing systems.

| Long-term recipient linkability | ✅ via Label Rotation per τ₁ | ✅ | ✅ | ✅ |
| Hosting third-party metadata | ⚠️ partial — inbox committee ≠ session node raises the account who-graph to role collusion; activity-timing / volume residual | ✅ via own node | ✅ | ✅ |
| Content surveillance | ✅ via Anchor encryption | ✅ | ✅ | ✅ |
| Quantum store-now-decrypt-later | ✅ via Noise_PQ XX (post-quantum transport end-to-end) | ✅ | ✅ | ✅ |
| Backbone GPS-precision timing-correlation | ⚠️ partial through aggregation (10⁶ msg threshold) | ⚠️ same + Tor obscures | ⚠️ + temporal randomization | ⚠️ + multi-path delays |
| Endpoint compromise (RAT) | ❌ out of scope | ❌ out of scope; mitigation via Light-Node-at-Home damage containment | ❌ | ❌ |
| Financial state visibility | ❌ public per [I-2] **by design** | ❌ same | ❌ same | ❌ same |

Markers: ✅ closed; ⚠️ partial / mitigated; ❌ not closed (intentional or a fundamental limit).

#### Endpoint compromise — Montana damage containment (architectural mitigation)

A network protocol cannot prevent endpoint compromise. **But Montana architecturally bounds the damage** through unique patterns:

- **Light-Node-at-Home split** (App spec, «Light-Node-at-Home»): the master_seed lives on the home node, the phone holds only ephemeral session keys. Compromising the phone ≠ compromising the master key.
- **Chain-anchored ephemeral session rotation per τ₁**: the session key is derivable via `HKDF(master_seed, current_window || "session-W")`. Maximum exposure window = 60 s.
- **Junona local pre-processing**: the AI agent on the own node performs decryption + summarization; the phone never holds the full content in memory.
- **Block-lattice sub-account hierarchy**: the phone uses a daily-spend sub-account ($X/day limit); the master savings account exists only on the home node.
- **Hardware-backed enclave integration**: the master_seed in the iOS Secure Enclave / Android StrongBox.

**Comparison with existing messengers:**

| System | Endpoint compromise impact |
|--------|----------------------------|
| Signal | Compromise device = full chat history forever (single trust domain) |
| WhatsApp | Compromise device = full history + cloud sync |
| Telegram | Compromise device = full history + cloud + saved messages |
| **Montana with Light-Node-at-Home** | Compromise phone = max loss `sub_account_limit × 60_sec_window_content` (multi-domain trust) |

#### What is **not** covered by any tier

Two fundamental open problems that **no one closes** (including Tor, Loopix, I2P):

1. **A global passive adversary with GPS-precision timing-correlation on all backbone links simultaneously.** Montana **uniquely weakens** it through canonical aggregation (a 10⁶-10⁸ message threshold instead of 10²-10³ as in Tor) but **does not close it absolutely** — an open research problem of the anonymity-network field since the 1980s.

2. **Endpoint compromise via RAT / hardware-level malware.** A network protocol is useless — the data is read before encryption. Montana **architecturally bounds the damage** through trust-domain split but does not **prevent** compromise. Full protection requires a hardware secure enclave + verified boot + careful endpoint hygiene.

These limits are **honestly recorded** in the spec — the user knows the scope of protection before use. The marketing claim of «total anonymity» is rejected as an overpromise that would collapse at the first serious audit.

### Seven layers — one construction

```
Layer 1: Transport Obfuscation          the personal server hides content and timing
Layer 2: Peer Selection                  start_window + network diversity constraints
Layer 3: NAT Traversal                   everyone can join, even behind NAT
Layer 4: Censorship-Resistant Discovery  six channels, one is enough
Layer 5: Dandelion++                     peers do not know who authored an operation
Layer 6: Mesh Transport                  operation during an internet shutdown,
                                         hop-by-hop Bluetooth / Wi-Fi Direct
Layer 7: Store-and-Forward Semantics     ephemeral buffering in the mesh,
                                         per-sender quota, signed acks
```

Each layer closes its own vector. None requires external infrastructure. Everything is built on top of libp2p (for internet layers 1-5) and native BLE / Wi-Fi Direct APIs (for mesh layers 6-7) plus the existing gossip. The network layer is orthogonal to consensus — no state transition is affected.

### Protocol Message Layer

Inside the IBT uniform frames, protocol messages follow a common envelope format. This section normatively defines the wire format of all Montana messages for cross-implementation compatibility.

**Envelope format.**

```
ProtocolMessage:
  msg_type         1B    <- u8, message type code
  msg_version      1B    <- u8, message format version (= 1 for the current wire format)
  request_id       8B    <- u64 little-endian, request/response correlation id (= 0 for one-way gossip)
  payload_length   4B    <- u32 little-endian, payload size in bytes
  payload          ?B    <- payload_length bytes, format determined by msg_type
```

The envelope is always a 14-byte header + payload. Because IBT uniform frames carry a 1021B payload, a ProtocolMessage may span one or several frames via the `0x04 continuation` flag (see Uniform Framing).

**Message type registry.**

| Code | Type | Direction | Payload |
|------|------|-----------|---------|
| 0x01 | Transfer | one-way gossip | Transfer object (Mode A or Mode B; serialized by canonical encoding; the mode is determined by payload length and the presence of the receiver in the Account Table) |
| 0x02 | reserved | — | Released (previously allocated to a separate activation gossip category; the gossip envelope namespace is independent of the operation type-byte). Do not re-allocate. |
| 0x03 | ChangeKey | one-way gossip | ChangeKey object |
| 0x04 | Anchor | one-way gossip | Anchor object |
| 0x10 | NodeRegistration | one-way gossip | NodeRegistration object |
| 0x20 | BundledConfirmation | one-way gossip | BundledConfirmation object |
| 0x21 | SSHA_Reveal | one-way gossip | SSHA_Reveal object |
| 0x22 | Proposal | one-way gossip | Proposal object |
| 0x40 | FastSyncRequest | request | `{anchor_window: u64, resume_offset: u64}` |
| 0x41 | FastSyncResponse | response | chunked snapshot data (see below) |
| 0x42 | FastSyncError | response | `{code: u8, message: bytes[≤255]}` |
| 0x50 | PeerListRequest | request | `{max_count: u16}` |
| 0x51 | PeerListResponse | response | `{count: u16, peers: count × PeerEntry}` |
| 0x60 | BatchLookupRequest | request | `{query_type: u8, count: u8, queries: count × query_entry}` (see the «Batch Lookup Protocol» section) |
| 0x61 | BatchLookupResponse | response | `{query_type: u8, count: u8, results: count × result_entry}` |
| 0x62 | BatchLookupError | response | `{query_type: u8, error_code: u8}` |
| 0xF0 | Ping | request | (no payload) |
| 0xF1 | Pong | response | (no payload) |
| 0xFF | Bye | one-way | `{reason: u8}` (graceful shutdown) |

Message versioning: `msg_version = 1` for the current wire format. A wire-format change increments msg_version and requires a protocol version upgrade.

An unknown msg_type → the receiver logs and ignores it (forward compatibility). An unknown msg_version → the receiver replies with FastSyncError, code `unsupported_version`, and closes the connection.

**Structured payloads.**

`PeerEntry`:
```
PeerEntry:
  ip_version       1B    <- u8, 0x04 or 0x06
  ip               16B   <- IPv4 in the last four bytes (first 12 = 0x00), or full IPv6
  port             2B    <- u16 little-endian
  node_id          32B
  start_window     8B    <- u64 little-endian, from the Node Table
= 59 bytes fixed
```

`FastSyncResponse` chunked delivery:
```
FastSyncResponse chunk:
  chunk_index      4B    <- u32 little-endian, starts at 0
  total_chunks     4B    <- u32 little-endian, total number of chunks for this request
  table_id         1B    <- u8: 0x01 Account, 0x02 Node, 0x03 Candidate, 0x04 Proposals
  record_count     4B    <- u32 little-endian, records in this chunk
  anchor_window    8B    <- u64 little-endian, the serving peer's head window the snapshot was assembled at
  records          ?     <- record_count × serialize(record) by canonical encoding
```

A response consists of N chunks (with one request_id). The receiver assembles by chunk_index. All chunks of one session must carry the same `anchor_window` (the serving peer's head); a chunk with a diverging anchor is rejected. After receiving all total_chunks the receiver rebuilds the Merkle root and compares it byte-for-byte with exactly `recent_roots[anchor_window]` — the cemented `state_root` the receiver independently observed through Proposal propagation at window `anchor_window`. A match is accepted, otherwise the snapshot is rejected. The receiver does not scan the whole set of observed roots for an arbitrary match — exactly the `anchor_window` from the response is authoritative.

**Connection lifecycle.**

Connection establishment order:

```
1. TCP SYN / SYN-ACK / ACK                (standard)
2. Noise_PQ XX handshake                   (3 messages: msg1=1184 B, msg2=7533 B, msg3=6349 B;
                                            mutual identity authentication is completed here)
3. IBT proof exchange                       (the client sends an ML-DSA-65 signature)
4. Access level determination               (node / candidate / account, see Transport Obfuscation)
5. Ready to exchange ProtocolMessages
```

Establishment timeouts (on the transport stack's local clock, outside the consensus-time scope; emergent values at genesis calibration):
- TCP connect: τ₁/2
- Noise_PQ XX handshake: τ₁/6
- IBT proof: τ₁/6
- All together no more than 1 τ₁ to readiness.

If any step exceeds its timeout → close, retry with another peer.

**Keepalive.**

- Ping once per τ₁ on an idle connection (no data).
- Pong must arrive before the current τ₁ ends at the Pong receiver.
- Three consecutive τ₁ without a received Pong → disconnect.
- During active data exchange Ping is not required (real data = evidence of activity).

Ping/Pong carry no payload — a pure liveness check. Any local RTT estimate by the sender is a concern of its OS transport layer (CLOCK_MONOTONIC, kernel-level, outside the consensus-time scope) and is not carried in any signed object.

**Graceful shutdown.**

The initiator sends Bye with a reason code:
```
0x00 — normal shutdown
0x01 — going offline for maintenance
0x02 — peer list refresh (attempt to find better peers)
0x03 — resource limits (too many connections)
0x04 — protocol violation (validation failed many times)
0x05 — version mismatch
```

The receiver acknowledges with its own Bye, then a Noise_PQ session close, then TCP FIN. At most τ₁/12 for a graceful shutdown (on the transport stack's local clock, outside the consensus-time scope), otherwise a forced close.

**Peer discovery algorithm.**

A new node at startup:

```
1. Extract discovery-seed peers from the built-in list (hard-coded).
2. Select 1-3 random discovery-seed peers, connect.
3. Perform IBT (the account keypair for a new node's first connection).
4. Send PeerListRequest with max_count = 128.
5. Receive PeerListResponse with up to 128 known peers.
6. Apply diversity constraints (/16, ASN, start_window) to the received list.
7. Select 24 outbound candidates by diversity.
8. Connect to the chosen ones in parallel.
9. After a successful IBT with a real peer, disconnect from the discovery seed (freeing seed slots).
10. Maintenance: PeerListRequest every ~τ₂ windows to refresh the "verified" peer table.
```

Discovery-seed exception:
- Limit: at most 3 concurrent connections to a discovery seed per node.
- Released after 13 real peers are connected.

**Peer exchange.**

Between two connected nodes:

```
Every τ₂_windows:
  A → B: PeerListRequest {max_count: 64}
  B → A: PeerListResponse {peers[]}
```

A node maintains two tables:
- **New** peers: recently learned (from bootstrap or PeerListResponse), not yet used.
- **Verified** peers: those with successful connections in the past.

When selecting outbound: 50/50 at random from both tables. Bucketing by the node's secret key prevents external enumeration.

**Retry policy.**

- Failed connect: exponential backoff in the transport stack's local clock seconds (1s, 2s, 4s, 8s, ..., max 5τ₁; outside the consensus-time scope).
- Peer rejected via IBT failure: the peer is marked bad for 1τ₁.
- Peer disconnected with reason 0x04 (protocol violation): the peer is blacklisted for 24τ₁.

**FastSyncError codes:**

```
0x01 snapshot_unavailable       -- the requested anchor_window is too old (the peer does not retain it)
0x02 snapshot_too_large          -- the snapshot is larger than the peer is willing to send
0x03 unsupported_version         -- msg_version is not supported
0x04 resource_exhausted          -- the peer is overloaded
0x05 access_denied               -- the peer does not serve Fast Sync to clients (nodes only)
```

**Network vs consensus — the boundary.**

Network-layer parameters (timeouts, retry delays, keepalive intervals) are implementation guidance and may vary between implementations with no consensus impact. The values in this section are recommended defaults. Consensus-critical: wire format (envelope, payloads), IBT proof format, message type codes. Changing a consensus-critical parameter requires a protocol version upgrade.

## Batch Lookup Protocol

The protocol provides **baseline privacy for lookup requests** for account-only users (those who operate through someone else's node without their own infrastructure). When a client requests information about accounts (a pre-key bundle, an existence check), the request is grouped into a batch of K elements, of which **exactly one** is the real target and the rest are random decoy accounts. The host sees a K-element request but does not know which position is real.

The mechanism applies only to cold-path lookups. The hot path — the user's already-known contacts — is resolved locally on the client without contacting the network (see the App spec «Contact discovery module» section).

### Constants

Defined in the Genesis Decree ProtocolParams:

- `batch_lookup_k = 16` — the mandatory batch size. Deviations are forbidden (for the derivation see the «Protocol constant derivation» section).
- `max_batch_lookups_per_τ₁ = 16` — a rate limit per account per τ₁ window, protecting the host from DoS.

### Message type 0x60 — BatchLookupRequest

Payload format:

```
BatchLookupRequest:
  query_type    1B   <- u8: 0x01 pre_key_bundle, 0x03 account_exists
  count         1B   <- u8, must == batch_lookup_k (= 16)
  queries       count × query_entry  (query_entry type depends on query_type)

where query_entry:
  query_type == 0x01 (pre_key_bundle):   32B account_id
  query_type == 0x03 (account_exists):   32B account_id
```

The client forms the batch: one real target + 15 decoy accounts, shuffled into an arbitrary order. The client remembers the real target's position within the batch locally.

**BatchLookupRequest invariants:**

- `query_type ∈ {0x01, 0x03}`; otherwise → reject `UnsupportedType` (error_code 0x02).
- `count == batch_lookup_k` (strict equality, = 16); otherwise → reject `InvalidCount` (error_code 0x03).
- The size of `queries[]` is exactly `count × entry_size(query_type)` bytes.
- The source account (the IBT-authenticated sender) is active in the Account Table; `max_batch_lookups_per_τ₁` is not exceeded in the current window.
- On rate-limit exceedance → reject `RateLimited` (error_code 0x01).

### Message type 0x61 — BatchLookupResponse

Payload format:

```
BatchLookupResponse:
  query_type    1B
  count         1B   <- = count from the request (16)
  results       count × result_entry  (in the same order as queries)

where result_entry:
  query_type == 0x01 (pre_key_bundle):   4B length prefix + variable ML-KEM-768 bundle
                                         (length=0 → bundle absent / never published)
  query_type == 0x03 (account_exists):   1B (0x00 → not found, 0x01 → exists)
```

The host **must** process all `count` queries and return `count` results in the same order. Partial answers are forbidden — either a full BatchLookupResponse or a BatchLookupError.

### Message type 0x62 — BatchLookupError

```
BatchLookupError:
  query_type    1B
  error_code    1B   <- 0x01 RateLimited, 0x02 UnsupportedType, 0x03 InvalidCount
```

### Host validation workflow

1. Verify the client's IBT authentication (level 3 accepted for account-only users).
2. Verify the structural invariants of BatchLookupRequest.
3. Verify the rate limit for the client account.
4. Perform `count` lookups against local state (Account Table, cemented Anchor archive).
5. Assemble `count` results in the same order as the queries.
6. Send the BatchLookupResponse.

The host does **not** log individual queries for privacy hygiene — only aggregate per-account rate counters for limit enforcement.

### Effective privacy analysis

At the target network scale of up to ~1B active accounts (an architectural goal; the `AccountRecord` state alone is ≈2.06 TB, and fast-sync at that scale is an open engineering benchmark) the client accumulates a passively-observed pool of active accounts through gossip proposals. Realistic pool size: 10K–100K accumulated over a τ₂ observation window.

At pool size 10K–100K and K=16:
- **Effective anonymity:** ~2–3 bits (1-in-4 to 1-in-8 practical protection).
- **Intersection-attack resistance:** an intersection attack requires ~1000+ observed batches (~decades of activity) — practically infeasible.
- **Semantic filtering:** the client must use per-function dummy pools (pre-key bundles only from accounts that published a bundle, and so on) — detailed in the App spec.

This is **partial protection**, not absolute. Full closure of the lookup surface comes through the own node (Light-Node-at-Home in the App spec). The protocol does the most possible under the constraints of [I-5] (commodity hardware, no PIR), [I-6] (no privacy mixers) and [I-7] (minimal cryptographic surface).

### Rate limit rationale

`max_batch_lookups_per_τ₁ = 16` at K=16 gives at most 256 queries per account per τ₁ window. Typical messenger user activity: ≤ 50 queries per session, several sessions per day. The limit covers reasonable usage and protects the host from DoS amplification.

On exceeding the limit the client receives a `RateLimited` error and must apply exponential backoff until the next τ₁ window.

### Invariant applicability

- **[I-5] Commodity hardware:** zero heavy crypto operations, only SHA-256 comparison for lookups — a standard read. Works on any commodity node.
- **[I-6] Regulatory compatibility:** a plaintext batch lookup is a bulk read operation, not a privacy mixer, ring signature, stealth address or hidden flow. The host sees all K queries explicitly.
- **[I-7] Minimal cryptographic surface:** zero new crypto primitives.
- **[I-15] Time-based scarcity:** rate limiting via `max_batch_lookups_per_τ₁` is a time-based defence, consistent with [I-15].
- **[I-16] Out-of-band identity binding:** batch lookup precedes the first message; the client receives a pre-key bundle, computes a fingerprint and shows it to the user for out-of-band verification. Compatible with [I-16] by construction.

### Local network tables — Storage Cards

The network layer creates local persistent tables outside consensus state ([I-3] is not violated — their derivation is local, entering no `state_root`). They are NOT covered by the «Storage Card per persistent table» section (which is about consensus state per [I-14]) but MUST have their own Storage Cards adapted for the local-state context: record size, growth model, lifecycle / eviction, hard quota, total disk usage. Without these cards the node operator does not know the disk requirements for the node runtime, and an implementation may ignore the quota and OOM on long operation.

The format is adapted for local state: cost-based fields are marked `n/a` uniformly (per the network layer's [I-15] rejection of monetary barriers), with protection through time-based / quota mechanisms.

**Storage Card — PeerRecord table**

```

                                        ≤ 1 PeerListRequest per τ₁ bounds the inflow rate)
```

### Network-layer binding KAT vectors

Each vector fixes a pair `(canonical input, expected byte-exact output)` for cross-implementation conformance per [I-9]. Inputs are fully integer-specified in the spec. Expected outputs are generated by the reference implementation (`mt-net::wire`) and filled in by a parallel spec patch — until then the vectors are in `conformance pending` status.

Generation methodology (for the reference implementation): run the encode/decode/sign function on the given inputs, capture the byte stream, hex-encode it, and replace each `pending` marker with the real hex in the spec in the same commit that adds the test fixtures to `mt-net::wire::tests`.

### Network layer — Threat Model

A mandatory normative section for the network layer. It enumerates adversary classes, protected properties, a coverage matrix (a protection mechanism per intersection), and an explicit out-of-scope list. Without this section Gate 11 (threat concentration) is not closed for the network mechanisms and an implementer cannot judge the sufficiency of the defences.

#### Adversary classes

| Class name | Capabilities | Budget (typical estimate) |
|------------|--------------|---------------------------|
| **Passive observer** | Observes encrypted traffic on links inside its own network (ISP, corporate network administrator, state SIGINT within a jurisdiction); analysis of timing, volume, metadata | Pervasive monitoring infrastructure |
| **Active MITM** | Inject / drop / delay / modify packets on links; false routing announcements (BGP hijack); rogue access points (Wi-Fi); compromised DNS resolver | Requires a network position (close to the victim or control of an AS) |
| **Eclipse attacker** | Controls ≥ ¾ of a target node's outbound peer slots by positioning its own nodes in peer tables; goal — isolate the victim from the honest network | Multiple registered nodes (×τ₂ windows of sequential SHA-256 per Sybil) + multiple IP /16 / ASN |
| **Sybil attacker** | Registers N formally legitimate nodes by passing the sequential-chain entry barrier repeatedly; goal — concentration of influence in peer selection / Dandelion stem / mesh forwarding | Linear cost per identity = τ₂ windows of sequential SHA-256 (N × τ₂); diversity constraints multiply the cost |
| **DoS attacker** | Flood: connection requests, frames, oversized payloads, invalid signatures, expensive operations; goal — exhaust the victim's compute / memory / disk / bandwidth | From a lone hobbyist (~10 Mbps) to a 100+ Gbps botnet |
| **Censor** | Blocks traffic to Montana nodes in its jurisdiction via DPI, SNI inspection, IP blocklist, port blocking; goal — denial of access to the network for the region's users | National-level firewall (Great Firewall, Roskomnadzor); commercial SaaS DPI |
| **State-level sabotage** | Sustained attack on storage / infrastructure through legitimate-looking operations; budget ≥ $1M; motivation = harm, not profit | Per the Gate 14 sabotage-actor budget model (parent role) |

#### Protected properties

| Property | Definition |
|----------|------------|
| **Confidentiality** | Message content is inaccessible to a non-recipient (including intermediate peers in Dandelion / SF / mesh) |
| **Integrity** | A message is not modified in transit; modification is detected |
| **Availability** | The node stays reachable for honest peers; consensus state keeps progressing |
| **Unlinkability** | An external observer cannot link operation X to the originating identity / IP of the sending node |
| **Identifier unlinkability (transport)** | The wire format of a Noise_PQ XX connection contains no long-lived identifier in the plaintext part — a passive observer cannot correlate two TCP sessions of one client across an application restart or a change of IP / network. Detailed analysis: [`External-Audit/transport-identifier-leakage.md`](External-Audit/transport-identifier-leakage.md). |

**Reachability-map poisoning (Censor / Sybil).** A hostile node reports false `ReachabilityObservation` records, claiming a dead or attacker-controlled entry is reachable, to steer honest nodes onto it (connectivity eclipse through steering). Closed by construction: a `(vantage_class, target_ref, profile)` triple is treated as reachable only when corroborated by observations from at least `REACHABILITY_QUORUM = 3` distinct /16 source groups — the diversity unit of the outgoing-connection constraints — and the final decision rests with the local IBT probe, not the map. The map is advisory: poisoning reorders candidate ranking and authorizes no connection. Status: **C**.

**Transport profile external dependencies (Censor).** Profiles T1, T2 and T3 introduce an external carrier the node does not control. T1 binds to a real cover host and relays any non-Montana connection to it, so an active prober sees a genuine reverse proxy; the host can go offline or change its TLS parameters and break the relay path, and the mitigation is cover-host rotation across a set of currently-reachable hosts and fallback down the ladder to T0. T2 routes through a content-delivery network: the provider can revoke the operator's tenant account and withdraw the carrier without notice; the mitigation is the availability of several CDN providers and fallback down the ladder to T1/T0. T3 routes through a pluggable-transport bridge: the bridge address can be enumerated and blocked by the censor, and the bridge operator is a trust party for carrier liveness — never for content, since the inner Noise_PQ XX session is end-to-end; the mitigation is bridge rotation through out-of-band distribution (Censorship-resistant discovery, channel 3) and fallback down the ladder. Neither profile weakens the inner post-quantum confidentiality or the identity binding; the dependency is on carrier availability, not on confidentiality. Status: **P** — availability rests on an external party, confidentiality is closed by construction.

**DNS-tunnelled bootstrap dependency (Censor).** Discovery channel 6 (DNS-tunnelled bootstrap) introduces a resolver-side responder as an external carrier of the same trust class as T2/T3: it delivers only a compact bootstrap peer record `(IP, node_id)`, never session data, and the inner Noise_PQ XX session is unaffected. DNS tunnelling is itself fingerprintable and DoH/DoT may be filtered; the mitigation is that the channel is one of six independent discovery channels, and it degrades to channels 1-3 and 5 where DNS tunnelling is blocked. Status: **P** — a bootstrap-only external carrier, confidentiality closed by construction.

**Volunteer transit-relay metadata surface (Sybil / Censor).** A volunteer transit-relay (see «Volunteer transit-relay role») observes both endpoints' IP addresses of the sessions it carries — never their content, which is end-to-end encrypted. An unbounded barrier-free relay pool would let one actor cheaply harvest a who-talks-to-whom graph at the IP layer (a Sybil on metadata rather than on consensus). Closed by construction through time, not money: a relay identity is eligible to carry transit only after `RELAY_SENIORITY_WINDOWS` of continuous reachability, and a served node distributes its relayed connections under the /16 and ASN diversity constraints, so no single actor's relays dominate a node's transit set. A censor cannot buy seniority, so a mass relay pool is expensive in the only resource Montana meters — time. Status: **C** — metadata-Sybil bounded by the [I-15] time barrier + diversity constraints.

**Entry-pool enumeration (Censor / Sybil).** A requester that asks many distributors for entry capsules seeks to harvest the whole pool and block it wholesale. Closed by construction: each responder returns only a bounded, requester-varied subset drawn by `Hash(responder_secret, requester_group)` under the peer-exchange rate limit (one message per τ₁ per peer), so coverage per vantage is bounded and a Sybil across many /16 groups still receives correlated, non-exhaustive subsets; blocking one subset neither reveals nor blocks the rest. Capsules are self-authenticating (ML-DSA-65 over address and freshness bound to `node_id`), so a hostile distributor can neither forge an entry nor extend a stale one. Status: **C** — pool enumeration bounded by pooled distribution + the [I-15] time-metered rate limit; capsule integrity closed by construction.
#### Out-of-scope (explicit list)

These threats are **not closed** by Montana's network layer, by design or by architectural decision:

1. **Endpoint compromise** — if a node's private key is stolen, the attacker fully impersonates the node on the network; the defence is operational (hardware key storage, OS hardening), not a network protocol.

2. **Traffic-analysis correlation at global scale** — an attacker controlling a significant fraction of all internet routers (Tier 1 ISPs, a state controlling a backbone exchange) can correlate timing across multiple connections and deanonymize even through Dandelion++. Defence requires a global-scale anonymity network (Tor with millions of relays), which is orthogonal to Montana.

3. **Side-channel attacks on crypto primitives** — timing / power / EM on ML-DSA-65 / SHA-256 implementations. Mitigation: use constant-time implementations (rustls / aws-lc-rs / ring); this is a [C-6] requirement, not a network-protocol concern.

4. **Quantum-capable adversary** — status: **closed**. PQ primitives (ML-DSA-65, ML-KEM-768) protect application auth and encryption (✅). The production transport is Noise_PQ XX (ML-KEM-768 ephemeral KEM on both sides + ML-DSA-65 identity signature + ChaCha20-Poly1305 AEAD); no classical Diffie-Hellman remains in the transport. Per [I-1] classical crypto is forbidden in the protocol layer; this invariant is honored end-to-end.

5. **Supply-chain attacks on dependencies** — a compromised library update (libp2p, rustls). Mitigation: Cargo.lock pinning + reproducible builds + dependency audit ([C-6] req #3-4-5-6); this is a build-time concern.

6. **Physical-proximity attacks on mesh** — BLE / Wi-Fi Aware require physical proximity (≤200m). An attacker with physical access can flood mesh advertisements. Mitigation: the per-sender rate-limit + IBT mesh proof requirement makes this nonproductive but not impossible (radio jamming is a physical-layer problem outside any application-layer defence).

7. **Legal/regulatory pressure on discovery-seed peers** — hard-coded discovery-seed peers can be seized / forced offline by a state. Mitigation: after the first connection a node operates through discovered peers; discovery seeds are needed only at cold start or after losing all peers; censorship-resistant discovery is a secondary path.

#### Backpressure rules — normative text

Applies to all IBT sessions + the ProtocolMessage envelope layer.

**Rule B1 — Pending requests cap.**

For each peer the count of active pending requests (awaiting response) ≤ `max_pending_requests_per_peer`. On an attempt to create a new request above the cap the caller receives `Error::Backpressure` and the request is not sent. After expiry (per `request_timeout_t1_div`) or a response the counter decreases.

**Rule B2 — Frame intake rate per peer.**

For each peer the incoming frame rate ≤ baseline_frame_rate × max_burst_factor, where `baseline_frame_rate = 1 fps` (see Uniform Framing) and `max_burst_factor = 8` (the max burst in Uniform Framing). On exceedance: drop excess frames silently (no disconnect — allows burst recovery), increment the peer-level penalty counter.

**Rule B3 — Total connection cap.**

A node maintains at most `max_outbound_per_node` outgoing + `max_inbound_per_node` incoming TCP connections at once. Outbound exceedance — the caller receives `Error::OutboundCapReached`, the attempt is refused until rotation frees a slot. Inbound exceedance — the incoming TCP connection is refused at the TCP level (RST) and the counter is not updated (protection against counter inflation).

**Rule B4 — Penalty escalation.**

The peer-level penalty counter increases on: (a) frame-intake-rate exceedance (B2); (b) a malformed payload after Gate 13a structural validation; (c) signature verification failure at IBT or on signed payload objects. On reaching the threshold = 100 events per τ₁ — disconnect the peer with reason 0x04 (protocol violation) + blacklist for 24·τ₁ (see «Retry policy»). The penalty counter resets at each τ₁ boundary or on a successful disconnect.

**Rule B5 — Resource-quota enforcement timing.**

All backpressure checks run **before** crypto verification (ML-DSA-65 verify ≈0.1 ms on a commodity CPU; protection against amplification attacks). Order:
1. TCP-level rate (OS firewall rules — outside spec scope).
2. Transport-level AEAD (rustls / aws-lc-rs — library scope).
3. IBT proof verify (ML-DSA-65 — inside the IBT card).
4. Frame structure parse (Gate 13a Uniform Framing invariants).
5. Frame intake rate per B2 — **before** ProtocolMessage decode.
6. ProtocolMessage envelope parse (Gate 13a).
7. Pending request cap per B1 — for request types.
8. Payload structure validation (Gate 13a per type).
9. Payload signature verify (if the payload is signed — Transfer / NodeRegistration / etc.).
10. Apply transition (apply_proposal or locally — outside backpressure scope).

**Rule B6 — Sabotage-actor protection.**

A sabotage actor with a $1M budget (per the Threat Model) and a realistic ≈10 Gbps resource can flood a node; standard upstream filtering (Cloudflare-like, ISP DDoS protection, hosting-provider mitigation) is out of spec scope and is expected operator practice. The spec backpressure rules protect only against **per-peer behaviour**, not against volumetric DDoS at the network level.

#### apply_mesh_frame and apply_store_and_forward — normative statements

The network apply-functions run on the local node's receive path. Unlike the consensus apply-functions (`apply_proposal`, `apply_emission`, etc.) they **enter no state_root** ([I-3] orthogonal — locally derived state). Their determinism nonetheless matters: reapplying the same frame must yield the same result (idempotency for replay handling).

##### apply_mesh_frame(frame, sender_pubkey, peer_local_state) → MeshIntake

```
Input:
  frame             MeshFrame (see the «MeshFrame wire format» section)
  sender_pubkey     ML-DSA-65 pubkey 1952B (obtained from the mesh advertisement)
  peer_local_state  &mut LocalMeshState {
    used_nonces:    Map<sender_pubkey_hash, Set<nonce>>,
    fragments:      Map<(sender_pubkey_hash, msg_id), Vec<Fragment>>,
    rate_window:    Map<sender_pubkey_hash, FrameWindow>,
  }

Output:
  MeshIntake = enum {
    Accepted,                    — frame validated, persisted or forwarded
    AcceptedComplete(payload),   — last fragment, message complete
    Rejected(reason),            — silent reject (no error to sender)
  }

Steps (ordered, deterministic):

  1. Structure validation per the MeshFrame invariants (9-field layout):
     - mesh_protocol_version ∈ {0x0001}
     - frame_type ∈ {0, 1, 2, 3}
     - ttl ∈ [0, 16] and hop_count ∈ [0, 16] and hop_count + ttl = 16
     - sender_ref exactly 32 B; recipient_hint exactly 32 B
     - payload_length ≤ 256; payload is exactly payload_length bytes
     - mac verifies under the link session key of the arrival hop
     Failed → return Rejected(InvalidStructure)

  2. IBT mesh proof verification per the «IBT mesh proof» card:
     - cached_window ∈ [known_W − 7·τ₁, known_W]
     - mesh_session_nonce ∉ used_nonces[hash(sender_pubkey)]
     - signature valid under sender_pubkey
     Failed → return Rejected(InvalidIBT)

  3. Frame intake rate per Backpressure Rule B2:
     - rate_window.intake_count++
     - intake_count ≤ baseline_frame_rate × max_burst_factor (1 fps × 8 = 8)
       within the latest 1 s window
     Exceeded → drop frame, increment penalty (Rule B4), return Rejected(RateLimit)

  4. Recipient determination:
     - recipient_hint == 0xFF × 32 → broadcast — addressed to all mesh peers
     - recipient_hint == hash(local_node_pubkey) → addressed to self
     - otherwise → addressed to another peer, a candidate for forwarding
       (see apply_store_and_forward for forwarding semantics)

  5. Fragment assembly (for multi-fragment messages; the fragment header
     lives in the ApplicationPayload, not the MeshFrame):
     msg_id = (sender_pubkey_hash, application fragment header)
     fragments[msg_id].insert(fragment_index, frame.payload)
     If fragments[msg_id].len() == total_fragments:
       payload_complete = concat(sorted by fragment_index)
       fragments.remove(msg_id)
       return AcceptedComplete(payload_complete)
     else:
       return Accepted

  6. Update used_nonces (after a full accept):
     used_nonces[hash(sender_pubkey)].insert(mesh_session_nonce)

Idempotency:
  apply_mesh_frame(F, S, state) twice:
    First call:   Accepted or AcceptedComplete(P), state' with the frame
                  registered.
    Second call:  Rejected(InvalidIBT) — the nonce is already in used_nonces, dedup.

  This is the manifestation of determinism: after the first accept a repeated
  frame with the same nonce is silently rejected, protecting against replay.
```

##### apply_store_and_forward(envelope, sender_pubkey, sf_local_state) → SFIntake

```
Input:
  envelope          SFEnvelope (see the «Buffer model» section)
  sender_pubkey     ML-DSA-65 pubkey (from envelope.sender_signature)
  sf_local_state    &mut LocalSFState {
    buffer:                Map<recipient_hint, BufferEntry>,
    sender_quotas:         Map<sender_pubkey_hash, SenderQuotaState>,
    recipient_acks:        Map<recipient_hint, SignedAck>,
    total_bytes:           u64,
  }

Output:
  SFIntake = enum {
    Buffered,                         — frame stored for forwarding
    DeliveredLocally(decrypted_msg),  — recipient = self, ML-KEM decrypt OK
    Rejected(reason),                 — silent reject
  }

Steps (ordered, deterministic):

  1. Structure validation per Gate 13a invariants for SFEnvelope:
     - recipient_hint exactly 32 B
     - ttl_window ∈ [current_W + 1, current_W + 24·τ₁]
       (TTL hard cap 24·τ₁; ttl_window ≤ current_W → expired, reject;
       ttl_window > current_W + 24·τ₁ → invalid, reject)
     - fragment_index ≤ total_fragments ≤ 255
     - sender_signature exactly 3309 B
     - ciphertext size ≤ max_sf_ciphertext_bytes (65 536, Genesis Decree)
     Failed → return Rejected(InvalidStructure)

  2. Sender signature verification:
     verify_message = canonical_encode(envelope_without_signature_bytes)
     ML-DSA-65 verify(sender_pubkey, verify_message, sender_signature)
     Failed → increment penalty (Rule B4), return Rejected(InvalidSignature)

  3. Per-sender quota per the «Per-sender quota» section:
     sender_quota = sender_quotas[hash(sender_pubkey)]
     sender_quota.frames_in_window_τ₁++
     If frames_in_window_τ₁ > 256:
       return Rejected(SenderQuotaExceeded) — silent (the sender infers it
       from the retry timeout)
     sender_quota resets automatically at each τ₁ boundary.

  4. Recipient ack check per the «Signed rate-limit acks» section:
     If recipient_acks[recipient_hint].is_revoked(sender_pubkey, current_W):
       return Rejected(ForwardingRevoked)

  5. Total buffer quota check per Storage Card sf_buffer:
     prospective_total = total_bytes + envelope_size
     If prospective_total > 1 GB AND sender NOT in the operator VIP whitelist:
       return Rejected(TotalQuotaExceeded)
     If prospective_total > 1 GB AND sender in the VIP whitelist:
       eviction: drop the oldest non-VIP entry until prospective_total ≤ 1 GB

  6. Recipient determination:
     If recipient_hint == hash(local_node_pubkey):
       attempt ML-KEM decrypt(local_node_secretkey, envelope.ciphertext)
       If success:
         remove envelope from buffer (delivered)
         return DeliveredLocally(decrypted_msg)
       If failure:
         return Rejected(DecryptFailure) — corrupted or unintended
     Else:
       buffer.insert(recipient_hint, envelope)
       total_bytes += envelope_size
       return Buffered

  7. Forwarding policy per the «Forwarding algorithm» section:
     After Buffered — the node periodically attempts forwarding:
     - selects peer candidates matching recipient_hint (or broadcast peers)
     - sends the envelope as a ProtocolMessage (an SF-reserved type in the
       extension namespace, wired at implementation time)
     - on successful delivery or TTL expiry → remove from buffer
       total_bytes -= envelope_size

Idempotency:
  apply_store_and_forward(E, S, state) twice with an identical envelope:
    First call:   Buffered or DeliveredLocally.
    Second call:  the per-sender quota is incremented again → may return
                  SenderQuotaExceeded; or, if the buffer already holds an
                  envelope with the same (recipient_hint, sender_pubkey,
                  ciphertext), `buffer.insert` is a replace, total_bytes is
                  unchanged, and it returns Buffered (idempotent on buffer
                  contents).

  Recipient ack revocation, sender-quota exhaustion, or TTL expiration make a
  repeated apply non-idempotent in the response but idempotent in the buffer
  state (either present, or expired/removed).
```

##### Local functions outside consensus state — why apply_*

The names `apply_mesh_frame` and `apply_store_and_forward` are symmetric to the consensus apply-functions (`apply_proposal`, `apply_emission`) deliberately: the **semantically identical role** — take input + state → produce new state. The difference is scope:

- consensus apply: state ∈ the consensus root (state_root computation).
- network apply: state ∈ the local node runtime (outside state_root, not consensus-critical).

This guarantees:

1. **Implementation pattern consistency** — the implementer knows that mesh / SF have the same semantics as a consensus apply: deterministic, ordered, idempotent (where specified).
2. **Cross-implementation conformance** — two independent implementations running the same frame through `apply_mesh_frame` obtain the same MeshIntake outcome. This is a conformance condition (not consensus-critical, but important for interop testing).
3. **[C-7] enforcement** — the code-side role prescribes no-shortcut on `apply_*` functions; mesh / SF now explicitly belong to this class, with no shortcut access to `used_nonces` / `buffer` / `sender_quotas` directly (only through apply).
