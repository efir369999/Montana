## Policy

**Single source of truth for the spec version is this file.** Code references the spec without a version pin (`// spec, section "<name>"`) — the version is always the current one from `Spec target:` above.

- Consensus-critical decisions in code are annotated with the comment `// spec, section "<name>"`. The version is not duplicated.
- On a spec bump (e.g. v29.6.8 → v29.7.0):
  1. Update this file: `Spec target:` + `Spec path:` + a new History entry.
  2. Rename the spec file (per the `feedback_spec_rename.md` rule).
  3. If a section in the spec was **renamed or removed** — walk the code (`rg "spec, section"`), update the specific references. If no sections were affected, **do not touch** spec comments in the code.
  4. One commit: `chore: spec bump vX → vY` with a description of what changed.
- Pre-mainnet breaking spec changes are applied immediately (see `CLAUDE.md` → Pre-mainnet).
